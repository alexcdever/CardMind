## 1. 目的与范围

本规格定义 CardMind 的产品定位、阶段目标与能力边界，作为产品层正式真相源。

本规格覆盖：

1. 产品终极目标与当前阶段目标。
2. 当前主目标用户与核心价值承诺。
3. 核心能力、能力边界与当前非目标。
4. 与现有正式规格之间的上下位关系。
5. 产品目标变更时的验收与演进规则。

本规格不覆盖：

1. 具体技术实现、API、存储结构与目录设计。
2. 营销文案、品牌包装或未进入正式约束的概念探索。
3. 分域行为细节；这些细节由下游正式规格继续定义。

## 2. 终极目标

基于当前审计结论，CardMind 的公开产品终极目标定义为：成为一款面向多设备个人用户、用于同步和流转笔记数据的应用。

该表述是当前仓库内正式化的产品方向收束结果；若后续采用等价措辞，MUST 保持“多设备个人用户”与“同步和流转笔记数据”这两个核心含义不变。

## 3. 当前阶段目标

当前阶段优先证明以下三件事：

1. 个人用户可在本地优先前提下稳定创建、编辑、查询笔记。
2. 同一用户跨设备同步笔记数据的体验可解释、可恢复、可长期信任。
3. 数据池协作能力作为同步与流转模型的扩展能力存在，但不得压过个人多设备主路径的优先级。

## 4. 目标用户

当前主目标用户 MUST 定义为跨平台个人用户——即同时使用 Android / Windows / macOS 中两种或以上系统、不把自己锁在单一生态内的个人用户。不得以团队管理者、组织管理员或复杂协作流程维护者为首要对象。

这里的核心场景是：用户在手机端（高频查看/低频记录）与桌面端（主力写作）之间持续延续同一批笔记数据，同步过程不依赖外网或云服务。

典型用户画像见 `docs/personas.md`。

## 5. 核心价值承诺

CardMind 当前对目标用户的核心价值承诺如下：

1. 本地优先：用户首先获得可直接写入与读取的个人笔记体验，不以在线状态作为记录前提。从点击图标到开始打字不超过 2 秒。
2. 纯粹与克制：系统不在写作过程中推送无关提示、推荐功能或捆绑其他云服务。用户打开应用的唯一目的是记录和查看笔记。
3. 低感知同步：同一用户跨设备同步时，系统 SHOULD 减少额外理解成本，让用户关注内容延续而非同步机制本身。同步只走局域网，不依赖外网或云存储容量。
4. 数据流转：笔记数据在 Android / Windows / macOS 设备间延续使用，不受单一生态绑定。
5. 可解释与可恢复：当同步出现异常时，系统 MUST 让用户理解当前状态，并提供恢复路径，从而建立长期信任。

这些承诺服务的核心不是"展示复杂技术能力"，而是帮助跨平台个人用户更稳定地保有、延续和流转自己的笔记数据。

## 6. 核心能力与能力边界

当前产品核心能力 SHOULD 收束为以下三类：

1. 个人笔记的创建、编辑、搜索与回顾。编辑器 MUST 完整支持 Markdown 语法（含中英文混排场景）。
2. 基于标签系统的内容组织——而不是仅靠文件夹归类。
3. 基于局域网（LAN）的跨设备低感知同步，不依赖外网或第三方云服务。

能力边界如下：

1. 网络同步限于局域网范围；外网或中继场景不属于当前阶段目标。
2. 标签是当前阶段唯一的内容组织方式；文件夹归类仅作为标签的辅助展示层。
3. 数据池协作是个人多设备主路径的扩展能力，不是更高优先级的独立主目标。
4. 任何新增能力若会削弱"个人用户先稳定记录，再可信同步"的主路径，SHOULD 视为偏离当前阶段目标。

## 7. 非目标

当前阶段的非目标包括：

1. 不以复杂团队协作作为第一优先级。
2. 不以多角色组织流程、组织治理或企业级审批链路作为第一优先级。
3. 不以扩张型平台化能力、开放生态或大而全功能覆盖作为第一优先级。
4. 多设备协同编辑（如光标感知、会话状态延续）不在当前阶段目标内，应延迟至个人笔记主路径在本地体验和跨设备同步稳定之后，再评估可行性。

若某项工作主要服务上述方向，而不能直接加强跨平台个人用户的笔记记录、查看和延续主路径，则 SHOULD 降低优先级。

## 8. 与实现的关系

本规格是产品层上位真相源。下游产品文档包括：

- `docs/personas.md` — 目标用户画像与核心使用场景（约束所有体验判断）

具体实现与行为细节由代码、测试与工程标准（`docs/standards/`）定义。历史设计与计划文档见 `docs/plans/`（仅供决策追溯，不视为当前真相源）。

当实现方向与本规格的产品目标、阶段目标、非目标约束冲突时，MUST 以本规格为准。

## 9. 验收与演进规则

1. 新的公开文档、下游规格与实现说明，MUST 不违背本规格定义的终极目标、当前阶段目标、目标用户与非目标。
2. 若产品目标发生变化，MUST 先更新本产品级真相源，再更新相关实现。
3. 若只是措辞调整而非方向调整，MUST 保持本规格中的核心语义不变，尤其不得改变“多设备个人用户”与“同步和流转笔记数据”的含义。
4. 任何超出当前阶段目标的新能力，SHOULD 先通过计划文档论证，再决定是否进入本规格与下游规格。
