## 决策记录

| 时间 | 决策 | 原因 |
|------|------|------|
| 2026-04-22 | invite 只能用于发起加入申请，不能直接让成员入池 | 用户明确确认“是否允许新成员加入”必须由管理员审核决定，invite 不应绕过正式审批链路 |
| 2026-04-22 | 数据池内任意成员都可以创建、查看并撤销 invite | 用户明确澄清 invite 管理边界不是 admin 专属，admin 的权限边界只体现在审批新成员加入 |
| 2026-04-22 | 单个 endpoint 在任一时刻只能属于一个未解散 pool，并且要在提交申请与审批阶段继续校验 | 仅在 create/join 入口校验不够，审批链路也必须守住同一业务不变量 |
| 2026-04-22 | Flutter 测试在质量脚本中固定使用 `-j 4` | 实测相对 `-j 1` 有明显收益，且当前 Flutter 测试可稳定并发执行 |
| 2026-04-22 | Rust 质量脚本固定使用 `cargo test --jobs 1` | Rust 的主要冲突层级在 Cargo 并发启动多个 test target 进程，而不是单个 binary 内部线程 |
| 2026-04-22 | 不再保留 `integration_exact` / `unit_exact` 两个重复测试入口 | 当前仓库没有现行脚本依赖它们，继续保留只会重复执行相同测试并放大 Rust 网络型不稳定 |
| 2026-04-22 | 不修改 `reset_app_config_for_tests()` 的正式语义，改为新增测试专用网络句柄清理口子 | 部分测试显式依赖“切换 app config 但保留 network handle”，直接扩展 reset 会引入行为回归 |

## 遇到的问题

- Rust/FRB 改动后主仓出现 content hash mismatch，需执行 `dart run tool/build.dart lib` 重建运行态 dylib 后再跑 Flutter 契约测试
- 当前环境下 `git add`、`git commit`、`git worktree remove` 等写操作需要提权执行，否则会遇到索引锁或权限限制
- Rust 网络集成测试的 `Failed to create netmon monitor` 根因不是 `iroh` API 用法错误，而是测试中残留 `PoolNetwork` 句柄未及时关闭，导致 monitor 资源逐步堆积
