# 移除引导页并改为卡片列表直达设计

## 1. 背景与目标

### 1.1 背景
- 当前产品首屏为引导页（`先本地使用` / `创建或加入数据池`），与新目标不一致。
- 用户希望进入应用后直接进入卡片笔记列表页，将数据池创建/加入行为收敛到 `Pool` tab。

### 1.2 目标
- 正式移除“引导页作为首屏”的产品规格与实现路径。
- 应用启动默认进入主壳，且默认分区为 `Cards`。
- 数据池入口仅保留在 `Pool` tab 的未加入态。
- 同步优化 `Pool` 页未加入态文案，明确“在这里创建/加入数据池”。

### 1.3 非目标
- 不改动池领域状态机语义（加入失败、同步异常、退出部分清理失败等）。
- 不新增卡片页中的池入口按钮。
- 不引入新的路由框架、全局导航服务或嵌套壳结构。

## 2. 方案对比与结论

评估方案：

1. 最小入侵：首页切壳，保留 onboarding 代码仅不再使用。
2. 中度重构（采用）：首页切壳 + 移除 onboarding 模块与文档引用。
3. 兼容过渡：默认切壳但保留隐藏 onboarding 入口。

采用方案 2 的原因：
- 规格与实现保持单一真相，避免“文档说无引导、代码仍有引导模块”造成长期歧义。
- 在改动规模与风险之间取得平衡：不触及池领域行为，仅清理引导相关资产。

## 3. 架构与导航设计

### 3.1 首屏入口
- `CardMindApp.home` 从 `OnboardingPage` 改为 `AppShellPage`。
- 首次启动直接进入 `AppShell`，默认 section 为 `cards`。

### 3.2 导航语义
- 保持当前主壳三分区：`cards / pool / settings`。
- 保持系统返回语义：
  - 若当前非 `cards`，返回先切回 `cards`。
  - 若当前已在 `cards` 根页，弹出退出确认。

### 3.3 引导模块处置
- 删除 onboarding 页面、控制器、状态及其引用链。
- 删除与 onboarding 首屏分流绑定的测试与文档条款。

## 4. 页面职责与交互

### 4.1 Cards 页
- 继续承担卡片主任务（搜索/列表/新建/编辑入口）。
- 不承载数据池创建/加入入口。

### 4.2 Pool 页
- `Pool` tab 成为唯一入池入口。
- 未加入态保留两个主动作：`创建池`、`扫码加入`。
- 未加入态文案新增或强化为“在这里创建或加入数据池”（语义等价文案可接受）。
- 已加入态、异常态、审批态、退池态保持现有行为契约。

## 5. 状态流与数据流

### 5.1 启动状态流
- 由 `Booting -> AppShellReady(cards)` 取代 `Booting -> OnboardingReady -> ...`。

### 5.2 池流程状态流
- 用户主动切换到 `Pool` tab 后进入池流程：
  - `PoolUnjoined -> JoinRequesting -> PoolJoined | JoinFailed(code)`
  - 退出链与异常恢复链维持既有定义。

### 5.3 数据边界
- 不引入新的跨页参数协议。
- 继续由 `PoolController + PoolState` 负责池侧状态推进。

## 6. 错误处理与回退策略

### 6.1 错误处理
- 保持现有错误分支与恢复动作：`retry`、`reconnect`、`重试清理`。
- 本次仅改入口与文案，不新增错误码与异常分层。

### 6.2 回退策略
- Git 提交保持单一意图拆分（文档/实现可独立回退）。
- 若后续产品验证需要恢复引导体验，应通过新规格变更重新引入，而非保留隐式旧入口。

## 8. 测试与验收设计

### 8.1 测试调整
- 新增/更新应用启动测试：首屏为 `AppShellPage` 且默认 `cards`。
- 删除 onboarding 相关 widget 测试。
- 回归 `AppShell` 返回链测试，确保双段返回语义不变。
- 更新 `PoolPage` 未加入态测试，覆盖文案强化与入口可见性。

### 8.2 验收标准（DoD）
- 仓内无 onboarding 运行路径与规格真相源。
- 启动即卡片列表页，且默认 tab 为 `cards`。
- 数据池创建/加入仅在 `Pool` tab 未加入态提供。
- 文档索引与规范门禁通过；Flutter 测试全绿。

## 9. 风险与缓解

- 风险：删除 onboarding 后，遗留引用导致编译或测试失败。
  - 缓解：先改首页入口，再删模块并执行全量引用清理。
- 风险：规格删除 S1 后与历史计划文档叙述不一致。
  - 缓解：本轮同步修订 specs 与必要 plan 索引描述，以 specs 为唯一验收真相。
- 风险：用户找不到池入口。
  - 缓解：在 `Pool` 未加入态强化文案并保持入口在首屏同一可视区。
