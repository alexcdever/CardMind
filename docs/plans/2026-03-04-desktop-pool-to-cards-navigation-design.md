# 桌面端池成功态到卡片列表导航闭环设计（2026-03-04）

- 日期：2026-03-04
- 状态：approved
- 关联规格：`docs/specs/ui-interaction.md`

## 1. 问题定义

在桌面端数据池创建成功后，用户停留在池页已加入态，当前页面缺少“回到卡片列表”的显式一步入口，导致主任务路径被阻断，违背“主路径可达”的交互目标。

## 2. 目标与边界

### 2.1 目标

1. 在桌面端 `PoolJoined` 视图中提供可见、可点击、一步到达的卡片入口。
2. 入口行为必须回归主壳导航语义（`AppSection.cards`），而非临时页面跳转。
3. 创建池成功后不再出现“只能留在池页”的交互死角。

### 2.2 非目标

1. 本轮不移除引导页。
2. 本轮不改动池协议、存储模型、同步协议。
3. 本轮不引入新的跨端 IA（信息架构）变更。

## 3. 方案选择

本设计采用“保留引导页 + 修复池成功态导航闭环”的最小修复方案：

1. 在 `PoolJoined` 视图增加“去卡片”主动作（按钮/显式入口）。
2. 该动作通过主壳 section 切换进入 `Cards` 工作区。
3. 保持现有 S1 引导双入口条款不变，避免触发整套规格重写。

## 4. 交互与状态语义

1. `PoolNotJoined -> createPool -> PoolJoined` 成功后，页面必须存在“去卡片”入口。
2. 用户点击“去卡片”后，目标状态为 `AppSection.cards`，且可见卡片主工作区（搜索、列表、新建入口）。
3. 从卡片返回池的路径仍通过主壳导航语义完成，保持双向可达。

## 5. 验收标准（黑盒）

1. Given 桌面端用户创建池成功，When 观察已加入态页面，Then 可见“去卡片”入口。
2. Given 用户在已加入态点击“去卡片”，When 导航完成，Then 一步进入卡片主工作区。
3. Given 用户已从池页进入卡片页，When 使用主壳导航返回池页，Then 池页状态与入口仍正常可达。

## 6. 风险与回滚

1. 风险：若使用页面内 `Navigator.push` 直接跳转，可能破坏主壳 section 一致性。
2. 规避：统一通过壳层 section 切换实现。
3. 回滚：若出现导航回归，可仅撤销“去卡片”入口与对应 section 切换逻辑，不影响池业务数据。
