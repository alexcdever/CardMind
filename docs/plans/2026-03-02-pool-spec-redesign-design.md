# 数据池规格重构设计（语言无关）

- 日期：2026-03-02
- 状态：approved
- 目标文件：`docs/specs/pool.md`

## 1. 背景

当前项目历史上以口头需求 + 设计/实现计划为主，关键产品约束分散在 `docs/plans/` 与代码中，导致：

1. 规格与实现边界不稳定，新增需求时容易重复解释。
2. 规格文档混入技术细节，难以跨语言与跨实现方式复用。
3. 变更时难以快速判断“行为是否变更”还是“仅实现替换”。

本次重构将 `pool` 规格改造成“可约束实现、语言无关、可验收”的正式规格样式。

## 2. 设计目标

1. `pool.md` 只描述数据池领域行为契约，不绑定 Flutter/Rust/FRB/SQLite 等实现。
2. 使用规范性措辞（MUST/MUST NOT/SHOULD/MAY）提高可测试性。
3. 保留对当前能力边界的真实表达：
   - 池元数据维护卡片笔记 `noteId` 引用集合。
   - 成员对卡片笔记增删改会最终同步给其他成员。
   - 当前仅要求 `loroDoc` 持久化文件同步，其他文件类型暂不纳入本规格。
4. 明确角色从 owner 统一为 admin，且创建池的 app 实例自动成为首个 admin。

## 3. 非目标

1. 不在本次规格中定义具体接口签名、类名、目录结构。
2. 不在本次规格中定义协议、分块传输、重传策略、存储引擎细节。
3. 不在本次规格中定义卡片笔记领域内部语义（如软删除字段结构）；该部分归属卡片笔记规格。

## 4. 关键决策

### 4.1 文档边界

- `pool.md` 覆盖：池生命周期、角色权限、不变量、同步可观察语义、错误语义、黑盒验收。
- `pool.md` 不覆盖：工程流程门禁、实现技术细节、测试命令。

### 4.2 角色模型

- 角色统一为 `admin` / `member`。
- 创建池时，创建者实例自动成为首个 `admin`。
- 未解散池必须至少存在 1 个 `admin`，任何导致 `admin=0` 的操作必须失败。

### 4.3 笔记与池关系

- 数据池元数据记录池内卡片笔记 `noteId` 集合（或等价引用集合）。
- 仅“创建卡片笔记”触发池侧元数据新增引用。
- 修改与删除在池侧都视为“引用对象状态变化”，不新增独立删除语义。

### 4.4 文件同步范围

- 本规格当前版本仅约束 `loroDoc` 持久化文件同步行为。
- 其他文件同步能力标记为 Out of Scope，后续由独立规格补充。

### 4.5 冲突与一致性

- 规格要求并发编辑可收敛为单一可解释终态，并具备可追溯性。
- 规格不限定具体冲突合并算法；当前实现可采用 Loro 或等价 CRDT 机制。

## 5. 备选方案与取舍

1. 方案 A（采用）：行为契约型重写。
   - 优点：语言无关、可测试、长期稳定。
   - 代价：初期需要从历史计划文档抽取并重述约束。
2. 方案 B：双层文档（规格 + 实现映射附录）。
   - 优点：迁移快。
   - 代价：实现映射易被误当规范本体。
3. 方案 C：最小修补原文。
   - 优点：改动小。
   - 代价：继续混入技术细节，不利于规格驱动。

## 6. 文档结构

重构后的 `pool.md` 采用以下主结构：

1. 目的与范围
2. 术语与角色
3. 领域对象与不变量
4. 生命周期与状态转移
5. 池内笔记与同步语义
6. 错误语义与恢复动作
7. 验收标准（Given/When/Then）
8. 版本边界与演进

## 7. 结果

本设计获批后，按以上结构重写 `docs/specs/pool.md`，并同步更新 `docs/specs/DIR.md` 与 `docs/plans/DIR.md` 索引。
