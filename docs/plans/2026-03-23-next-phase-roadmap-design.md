# 下一阶段 roadmap 设计

**日期**: 2026-03-23  
**状态**: 已确认设计，待撰写实施计划  
**作者**: OpenCode

---

## 1. 背景

`docs/specs/product.md` 已明确 CardMind 当前公开产品方向：面向多设备个人用户，以本地优先、低感知同步和可解释的数据流转支撑笔记数据的长期使用与跨设备可信流转。

`docs/specs/user-journeys.md` 进一步把这一方向下沉为五段主路径旅程：首次理解、开始记录、跨设备延续、异常与恢复、长期使用与信任稳定。

在此基础上，当前需要的不是新的功能散点清单，而是一份能够回答“下一阶段先证明什么、后证明什么”的产品优先级导向 roadmap 设计。

该 roadmap 必须避免以下偏移：

1. 重新滑回按功能罗列的实现任务单。
2. 让数据池协作、设置补全或平台化扩张压过个人多设备主路径。
3. 把“异常恢复”视为主路径之外的边缘补丁。

因此，本设计的核心任务是基于现有两份真相源规格，为 CardMind 定义一个两层结构的下一阶段产品路线判断。

---

## 2. 目标

本设计的目标不是产出版本功能表，也不是直接拆实现任务，而是回答以下问题：

1. CardMind 下一阶段的唯一主目标应该优先证明什么。
2. 哪些能力应作为配套约束一起进入该阶段，而不是延后补救。
3. 哪些看似合理的方向当前不应进入主优先级。
4. 在近期 phase 之后，后续 1-2 个 phase 应如何顺序推进。

这份 roadmap 设计应服务产品优先级判断，而不是替代后续 implementation plan。

---

## 3. 规划原则

本次 roadmap 设计 MUST 同时受以下原则约束：

1. 主路径优先：个人多设备笔记主路径优先于协作扩展、设置补全或平台化扩张。
2. 先延续成立，再扩能力：当前阶段应优先证明“同一份笔记数据可在多设备上连续使用”，而不是先展示更多能力。
3. 异常恢复属于主路径：凡是会削弱用户对内容安全、状态理解和下一步动作判断的缺口，都不应被视为后置修补项。
4. 可信任高于完整性幻觉：当前 roadmap 不应为了看起来更完整而补齐低价值入口或空壳能力。
5. 扩展能力后置评估：数据池协作与其他数据流转扩展能力，只有在个人多设备主路径已相对稳定后，才应重新评估优先级。

---

## 4. 当前判断

基于 `docs/specs/product.md` 与 `docs/specs/user-journeys.md`，本设计作出以下判断：

### 4.1 近期唯一主目标应为“跨设备延续成立”

原因如下：

1. `docs/specs/product.md` 已将“多设备个人用户”与“同步和流转笔记数据”定义为产品终极方向中的核心语义。
2. 当前阶段三大证明目标中，最能体现 CardMind 与普通本地笔记或普通云笔记差异的，就是“同一用户跨设备同步笔记数据的体验可解释、可恢复、可长期信任”。
3. 如果用户在第二台设备上仍把体验理解为重新导入、重新配置或重新开始，那么产品主价值仍未真正成立。

因此，下一阶段 roadmap 不应继续以泛化的“可用性补齐”或“功能扩张”作为主轴，而应明确聚焦：让用户在另一台个人设备上把自己的笔记经历理解为同一条连续主路径。

### 4.2 “异常与恢复”必须作为该阶段的配套约束

虽然近期 phase 的唯一主目标是“跨设备延续成立”，但 `docs/specs/user-journeys.md` 已明确异常恢复属于主路径的一部分。

这意味着：

1. 只追求“成功路径上看起来能同步”，但异常时状态不可理解、内容安全边界不清、下一步动作不明确，不足以支撑该阶段成立。
2. 近期 phase 中所有与跨设备延续相关的产品判断，都必须同时回答三个问题：
   - 用户是否理解当前看到的是不是同一份数据。
   - 用户是否知道内容是否安全。
   - 用户是否知道接下来该做什么。

因此，“异常与恢复”在近期 phase 中不是并列主目标，但必须作为进入该阶段的配套约束被同时设计和验证。

这里需要明确分层：

1. **Phase 1 只要求最低恢复能力成立**：用户在跨设备延续出现波动时，至少能判断当前是否还是同一份数据、内容是否安全、下一步该做什么。
2. **Phase 2 才要求系统性恢复能力成立**：用户不仅能勉强继续使用，还能在更稳定、可预期的状态表达和恢复路径中保持信任。

换句话说，Phase 1 解决的是“不要在异常出现时立刻断路”，Phase 2 解决的是“让异常与恢复本身也成为可信赖的连续体验”。

为了便于后续 planning 拆分，二者的判定边界进一步定义为：

1. **最低恢复能力**：只覆盖轻度波动或单步阻塞场景，目标是让用户不误判为“内容没了”或“必须重新开始”。
2. **系统性恢复能力**：覆盖重复出现、状态持续不明或需要明确恢复动作的场景，目标是让用户在更复杂的异常下仍保持稳定预期。

### 4.3 以下方向当前不应进入主优先级

当前不应进入主优先级的方向包括：

1. 单独建设“最小可信设置中心”。
   - 延后原因：当前它不能比“跨设备延续成立”更直接证明产品核心价值。
   - 重开条件：只有当设置能力已明确成为主路径理解、延续或恢复的必要支点时，才值得单独立项。
2. 将数据池协作重新抬升为并列主线。
   - 延后原因：这会分散个人多设备主路径的注意力，并削弱当前阶段的价值证明聚焦。
   - 重开条件：只有当个人多设备主路径已形成稳定可信赖的连续体验后，才应重新评估协作扩展优先级。
3. 围绕完整性幻觉去补齐更多非主路径入口。
   - 延后原因：这类工作容易制造“看起来更完整”但不增强主路径成立的假进展。
   - 重开条件：只有当某个入口明确承担主路径信息澄清或恢复动作时，才应重新讨论其必要性。
4. 为了版本包装提前推高长期能力或平台化外延。
   - 延后原因：长期信任是结果性目标，平台化外延更是更晚层级的扩展，不应先于主路径成立。
   - 重开条件：只有当 Phase 1 和 Phase 2 的目标都已基本成立后，才应讨论更大版本叙事。

这些方向并非永远不做，但在当前阶段，它们都不能比“跨设备延续成立”更优先。

## 5. 两层 roadmap 结构

本次推荐采用“两层结构”的 roadmap：先定义近期一个主 phase，再给出后续两个 phase 的方向性安排。

### 5.1 Phase 1：证明跨设备延续成立

#### 阶段目标

让用户在第二台个人设备上继续看到、读取并延续同一份笔记数据时，把这段体验理解为同一条连续主路径，而不是一次迁移、导入或重新开始。

#### 优先事项

1. 强化“同一份数据”的连续感知，让跨设备使用被理解为延续而非重建。
2. 在主路径关键节点提供最低必要的状态理解能力，使用户能够判断当前状态、内容安全边界和下一步动作。
3. 继续移除会打断“记录 -> 延续 -> 继续记录”理解链条的主路径噪音。

#### 明确不做

1. 不把设置中心建设列为该阶段主目标。
2. 不把数据池协作扩张为并列产品主线。
3. 不为了展示能力而增加与当前主目标无直接关系的功能面。

#### 必须覆盖的异常范围

Phase 1 只覆盖以下代表性场景，用于定义最低恢复能力边界：

1. 轻度同步波动：用户暂时无法确认最新内容是否已在另一台设备可见。
2. 状态短暂不明：系统处于处理中或等待中，用户需要知道此刻是否仍可继续记录。
3. 单步延续受阻：用户在切换设备后没有立即看到预期内容，需要判断当前应等待、重试还是返回原路径确认。

以下场景不要求在 Phase 1 内系统性解决，而应进入 Phase 2：

1. 重复出现的异常模式。
2. 需要明确恢复流程编排的场景。
3. 会显著影响长期信任判断的复杂异常体验。

#### 成立信号

1. 用户在另一台设备上看到内容后，自然理解自己是在继续同一份笔记数据。
2. 即使状态有波动，用户仍不会立刻失去方向感或把体验理解为重新开始。
3. 用户对产品的核心认知收束为“这是一个能让我在多设备间继续同一份内容的笔记产品”。

#### 建议验证口径

1. 任务 walkthrough 中，用户能用自己的语言复述“这是同一份内容的继续使用”，而不是迁移或导入。
2. 在包含轻度状态波动的演示或测试脚本中，用户仍能指出内容是否安全以及下一步动作。
3. 产品评审或可用性检查时，近期变更都能被解释为在增强“延续成立”，而不是仅增加功能面。

#### 进入下一阶段的条件

只有当跨设备延续已经不再主要依赖用户自行脑补，并且异常时基础状态理解已经能支撑继续使用，才进入下一阶段。

### 5.2 Phase 2：把延续变成可解释、可恢复、可依赖

#### 阶段目标

把“偶尔可延续”提升为“在出现异常、波动和阻塞时，用户仍能判断内容是否安全、当前状态是什么、下一步该做什么，并愿意继续使用”。

该阶段是在 Phase 1 最低恢复能力已成立的前提下，继续补强更复杂、更持续的异常与恢复体验。

#### 优先事项

1. 将异常与恢复从最低可继续使用的底线，提升为更稳定、可预期的正式主路径组成部分。
2. 让同步与延续过程中的状态表达更稳定、更可理解。
3. 让恢复动作不再依赖用户自行猜测系统意图。
4. 让一次异常不会直接摧毁用户对长期使用的信任。

#### 明确不做

1. 不急于扩张更多玩法或协作外延。
2. 不以“功能更多”作为此阶段的价值证明方式。

#### 成立信号

1. 用户遇到异常后，仍能判断内容安全边界并采取下一步行动。
2. 异常不会直接把用户排除出继续记录或继续延续的主路径。
3. 用户开始形成对系统状态的稳定预期，而不是每次都重新猜测。

#### 建议验证口径

1. 在异常场景 walkthrough 中，用户能够独立说出当前状态、恢复动作与内容安全边界。
2. 对同类异常场景的多次演练中，用户判断结果基本一致，不再依赖临场解释。
3. 后续计划中的恢复相关工作项，能明确区分状态表达、恢复动作与信任建立三类目的。

#### 进入下一阶段的条件

只有当“延续 + 恢复”已经形成基本可信赖的连续体验，才应评估更外层的数据流转扩展能力。

### 5.3 Phase 3：谨慎评估数据流转扩展能力

#### 阶段目标

在不压过个人多设备主路径的前提下，重新判断哪些数据池协作或其他流转扩展能力，真正能够强化“笔记数据持续流转”而不是制造新的理解负担。

该阶段首先是一个 `优先级重评估 phase`，而不是默认进入大规模实现的功能扩张 phase。

其默认输出是：继续 defer、单独设计，或进入后续规划；而不是直接形成新的实现大清单。

#### 优先事项

1. 只引入那些能服务个人主路径延续和流转的扩展能力。
2. 维持“主路径优先、扩展能力后置”的产品边界。

#### 明确不做

1. 不让协作扩展反客为主。
2. 不把产品重新定义为以复杂协作为核心的系统。
3. 不默认把任何扩展能力直接升格为近期实现承诺。

#### 成立信号

1. 扩展能力被用户理解为“帮助我的内容继续流转”，而不是又一套需要先学习的新系统。
2. 个人多设备主路径仍保持最清晰、最可信的产品中心。

#### 进入下一阶段的条件

只有当相关扩展能力已经通过优先级重评估，被证明不会削弱个人主路径理解与信任时，才应进入新的专项设计或实施计划。

#### 建议验证口径

1. 对每个候选扩展能力，都能明确说明它服务主路径的哪一段，而不是只说明它“功能上有用”。
2. 若移除某项候选扩展能力，个人多设备主路径仍然成立；若保留它，则主路径被进一步增强而不是被稀释。
3. roadmap 评审中，Phase 3 的输出首先是“继续 defer / 单独设计 / 进入后续规划”的决策，而不是直接排实现期。

---

## 6. 优先级边界

为了防止 roadmap 漂移，本设计明确以下优先级边界：

### 6.1 不为完整性幻觉补设置中心

设置相关能力只有在它明确服务当前主路径理解、延续或恢复时，才值得进入后续设计讨论。仅仅因为当前产品“看起来应该有设置”，不足以成为近期 roadmap 的优先事项。

### 6.2 不为展示能力推高数据池协作

数据池协作在当前阶段的定位仍然是服务个人多设备主路径的数据流转扩展能力，而不是更高优先级的独立主目标。任何会削弱个人主路径注意力的协作扩张，都应降低优先级。

### 6.3 不为版本包装提前堆长期能力

长期信任稳定是结果性目标，不应在跨设备延续尚未站稳时，被包装成一个功能面更大的近期版本承诺。正确顺序应是：先让延续成立，再让恢复稳定，最后验证长期依赖是否成立。

## 7. 对后续 planning 的约束

本设计完成后，后续 implementation plan 应遵循以下约束：

1. 不直接按页面或工程模块拆，而应先围绕“跨设备延续成立”这一产品判断进行任务化分解。
2. 所有近期工作项都应能回答：它是在增强“同一份数据的连续感知”、增强“异常时的状态理解”，还是在移除主路径噪音。
3. 若某项工作不能直接服务上述三类目标，则不应进入近期 phase 的主清单。
4. 对“最小可信设置中心”与“数据池扩展能力”的任何推进，都应作为后续独立设计议题重新论证，而不是默认并入当前阶段。
5. implementation plan 必须显式产出一份 `defer 清单`，并为每项 defer 记录：延后原因、重开条件、所属后续阶段。

### 7.1 实现分层约束

由于当前 phase 讨论的是“跨设备延续成立”与“最低恢复能力成立”，后续 implementation plan MUST 显式遵守以下架构落地边界：

1. Rust 是该阶段的主实现层，负责输出同步状态、错误语义、恢复动作边界与“是否仍是同一份数据”的稳定契约。
2. FRB 只负责跨语言传输 Rust 已定义的稳定 DTO、错误码与状态语义，不应在边界层新增独立业务判断。
3. Flutter 只负责展示这些后端契约、编排页面交互与验证 UI 可感知结果，不应复制 Rust 已掌管的同步/恢复业务判断。
4. 若某个近期工作项的核心价值在于改变同步、恢复或连续性语义，则该工作项默认应先落在 Rust 侧，再决定 Flutter 是否需要做展示对齐。
5. 只有当某项工作纯属导航收口、文案表达或可见反馈对齐时，才可以只修改 Flutter / 文档层而不涉及 Rust。

### 7.2 Planning handoff 输出物

为了保证 roadmap 向 implementation plan 的交接完整，后续 planning 至少应显式产出以下四类输出物：

1. `近期主清单`：只包含服务 Phase 1 主目标的工作项。
2. `defer 清单`：列出当前不进入近期 phase 的事项，并写明延后原因、重开条件、所属后续阶段。
3. `验证清单`：列出用于判断“跨设备延续成立”与“最低恢复能力成立”的 walkthrough、评审或测试脚本。
4. `阶段外议题清单`：列出需要单独发起设计的问题，例如最小可信设置中心、数据池扩展能力重评估。

---

## 8. 后续步骤

在本设计确认后，后续步骤应为：

1. 基于本设计撰写下一阶段 implementation plan。
2. 将 implementation plan 的近期工作项收束到“跨设备延续成立 + 异常恢复配套约束”。
3. 在后续计划中保留两个显式待判断问题：
   - 未来是否需要单独立项“最小可信设置中心”。
   - 哪些候选数据池扩展能力满足进入 Phase 3 重评估的前提。
4. 待 implementation plan 明确后，再进入具体规格补齐或实现任务拆解。
