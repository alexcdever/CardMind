# 2026-03-16 Rust 多入口后端化设计

## 1. 背景与目标

- 当前仓库已经把项目级长期方向收敛为：Flutter 作为前端，Rust 作为后端，`LoroDoc` 作为唯一真实信源，`SQLite` 作为查询侧读模型。
- 现状中的 Rust 后端能力虽然已经服务于 Flutter/FRB 主路径，但对外形态仍然偏向“面向 FRB 的内嵌后端门面”，尚未收敛为可被多入口复用的统一后端内核。
- 本设计的目标是把 Rust 进一步后端化：不只面向 Flutter 提供 FRB，还要面向本机 HTTP、MCP 与调试 CLI 提供统一能力，同时保持同一套领域规则、读写链路和稳定契约语义。
- 第一阶段目标不是做远程部署平台，也不是优先扩展云端能力，而是在当前 Flutter App 项目形态下，建立“单 Rust 后端内核 + 多入口适配层”的稳定架构。
- 第一阶段同时要求四类入口围绕同一批最小完整核心用例形成闭环，并把“多入口契约一致”作为首要验收目标。
- 本设计不仅涉及功能能力扩展，也涉及正式规格文档的同步演进：如果当前 `docs/specs/` 对 Rust 后端边界、入口能力或运行时行为的正式约束不足，则后续实施前必须先完成规格文档 CRUD 与一致性更新。

## 2. 已锁定决策

- Rust 是唯一后端内核，`FRB`、`CLI`、`HTTP`、`MCP` 都只是入口适配层。
- `FRB` 仍然是 Flutter App 默认主入口，默认启用。
- `CLI`、`HTTP`、`MCP` 默认不启用。
- 可选入口的启用状态由 Rust 侧配置文件持久化管理，而不是由 Flutter 临时页面状态决定。
- `HTTP` 第一阶段仅监听 `127.0.0.1` / `localhost`，但设计上预留未来改为可配置监听地址的演进空间。
- `MCP` 的定位是 AI/Agent 工具入口，而不是对现有 HTTP API 的简单别名。
- `CLI` 的定位是调试/运维入口，不是第一阶段面向普通终端用户独立分发的外部产品。
- `HTTP` 与 `MCP` 在启用后必须在 App 运行期间持续可达。
- `CLI` 启用后必须在 App 运行期间作为受控调试入口可用。
- 四个入口必须共享同一批核心用例语义、DTO 核心语义、错误码与状态语义。
- 本设计的后续 implementation plan 必须把正式规格文档更新纳入首批任务，而不能只做代码重构。

## 3. 备选方案比较

### 3.1 方案 A（选定）：共享应用服务层 + 各入口独立适配层

- Rust 内部分层为后端内核与入口适配层：后端内核负责领域规则、应用服务、存储与运行时；`FRB`、`CLI`、`HTTP`、`MCP` 分别做各自协议适配。
- 优点：
  - 最符合“Rust 彻底后端化”的目标；
  - 最容易保证多入口契约一致；
  - 能避免把 FRB、HTTP 或 MCP 的协议形状误当成真实后端接口；
  - 有利于后续继续扩展入口而不复制业务规则。
- 缺点：
  - 需要一次明确的模块重整；
  - 需要补齐统一应用服务、配置模型与运行时管理层。

### 3.2 方案 B：以 HTTP API 为中心，CLI/MCP 尽量复用 HTTP

- 先做本机 loopback HTTP 服务，再让 CLI 与 MCP 主要建立在 HTTP API 之上。
- 优点：
  - 外部观察到的接口形状更统一；
  - 调试和抓包较方便。
- 缺点：
  - 会把本机内核能力过早绑死在 HTTP 资源表达上；
  - CLI 与 MCP 容易退化为 HTTP 套壳；
  - 不利于保持 MCP 的工具语义。

### 3.3 方案 C：在现有 FRB API 基础上继续外挂入口

- 保持当前 `rust/src/api.rs` 的主导地位，再逐步向外追加 CLI/HTTP/MCP 能力。
- 优点：
  - 短期改动最小；
  - 起步最快。
- 缺点：
  - 最容易把“面向 FRB 的门面”继续误演化为唯一后端接口；
  - 长期会放大历史命名与分层包袱；
  - 难以从根上证明 Rust 已成为面向多入口的后端内核。

## 4. 顶层架构与模块边界

### 4.1 顶层架构

- 目标架构收敛为：`Flutter UI -> FRB`、`调试入口 -> CLI adapter`、`本机集成方 -> HTTP adapter`、`AI/Agent -> MCP adapter`，四类入口都进入同一个 Rust 后端内核。
- Rust 后端内核内部按职责拆分为：
  - `domain`：领域对象、领域规则、不变量；
  - `application`：面向用例的稳定服务接口、DTO、统一错误与状态语义；
  - `infrastructure`：`LoroDoc`、`SQLite`、同步网络、路径、配置文件、底层存储与技术实现；
  - `runtime`：入口启停、运行时资源持有、健康状态、配置驱动启动与关闭。
- 入口层只承担协议适配和调用转换职责，不复制业务规则，不直接改写存储事实。

### 4.2 入口职责

- `FRB`：Flutter 的默认主入口，负责语言边界上的请求、响应与错误传递。
- `HTTP`：本机 loopback API 入口，面向本机其他应用或工具调用。
- `MCP`：AI/Agent 工具入口，负责把应用服务暴露为工具语义，而不是简单复制 HTTP 路由。
- `CLI`：受控调试/运维入口，便于开发、诊断、验证和回归。

### 4.3 运行期总原则

- App 启动后，Rust 后端初始化读取配置，并决定是否启动 `HTTP`、`MCP`、`CLI`。
- `FRB` 作为默认主路径，总是可用。
- 已启用的 `HTTP`、`MCP`、`CLI` 在 App 运行期间必须保持可访问或可调用，不能只在配置层面标记为 enabled 却没有实际运行实例。

## 5. 统一用例面与契约一致性

### 5.1 应用服务层是唯一正式后端接口

- Rust 必须先定义“应用服务层”作为唯一正式后端接口。
- 真正稳定的后端接口应是用例语义，而不是 FRB 函数签名、HTTP 路由形状、CLI 命令格式或 MCP tool 名称。
- 各入口都必须通过应用服务层访问后端能力，而不能直接耦合到底层 store、network 或历史门面函数。

### 5.2 第一阶段共享的最小核心用例集

- 后端初始化与配置读取。
- 入口启用配置读取与更新。
- Pool：创建、加入、列表、详情。
- Card：创建、更新、列表、详情。
- Sync：状态查询、手动同步。
- 基础健康/就绪状态查询。

### 5.3 共享什么，不共享什么

- 四入口必须共享：
  - 用例语义；
  - DTO 核心字段；
  - 错误码；
  - 成功/失败/降级状态语义；
  - 读写链路约束与验收行为。
- 四入口不必完全共享：
  - 参数承载方式；
  - 协议包装形式；
  - 输出格式细节；
  - 交互风格。
- 例如同一个 `createCardNote` 用例可以分别表现为 FRB 调用、HTTP 路由、CLI 命令或 MCP tool，但它们都必须命中同一应用服务，并得到同一业务语义。

## 6. 配置文件与入口运行时设计

### 6.1 配置模型

- Rust 后端维护自己的配置文件，并把其视为后端事实的一部分。
- 配置中至少新增以下字段，默认值均为 `false`：
  - `http_enabled`
  - `mcp_enabled`
  - `cli_enabled`
- `FRB` 不参与开关模型，因为它是 Flutter App 的默认主路径。
- 如果未来需要把 HTTP 监听地址开放为可配置项，应作为同一配置模型的演进，而不是另起一套配置通道。

### 6.2 配置与运行时收敛

- Rust 初始化时读取配置文件，并据此启动对应入口。
- 当配置变更时，Rust 不仅要写回配置文件，还必须让当前运行时向目标状态收敛：
  - 启用 `HTTP` 时，当前会话中真正启动 HTTP 监听；
  - 关闭 `HTTP` 时，当前会话中真正停止监听；
  - 启用 `MCP` 时，当前会话中真正启动 MCP 服务；
  - 关闭 `MCP` 时，当前会话中真正停止 MCP 服务；
  - 启用 `CLI` 时，当前会话中真正暴露受控调试入口；
  - 关闭 `CLI` 时，当前会话中真正撤销该入口。

### 6.3 常驻入口约束

- `HTTP` 启用后，必须在 App 运行期间保持本机可达。
- `MCP` 启用后，必须在 App 运行期间保持可连接、可发现或可调用。
- `CLI` 启用后，必须在 App 运行期间保持受控调试入口可用。
- 因此 Rust 需要统一的运行时入口管理器来持有各入口句柄、健康状态、错误状态和关闭逻辑。

### 6.4 HTTP 的当前与未来边界

- 第一阶段 HTTP 监听地址硬编码为 `127.0.0.1` / `localhost`。
- 第一阶段不提前引入外网访问、远程部署、安全鉴权或公开服务暴露能力。
- 但正式设计应明确：HTTP 监听地址属于入口配置的一部分，未来可扩展为可配置字段。

## 7. DTO、错误与状态语义

### 7.1 DTO 设计原则

- DTO 应先在应用服务层定义核心稳定语义，再由各入口根据协议需要做薄包装。
- 应用服务层共享 DTO 的例子包括：
  - `PoolDto`
  - `PoolDetailDto`
  - `CardNoteDto`
  - `SyncStatusDto`
  - `BackendConfigDto`
  - `RuntimeEntryStatusDto`
- 入口层包装 DTO 仅用于协议输出，不得改写业务字段含义。

### 7.2 错误契约

- 统一错误契约继续以稳定 `error.code` 为核心。
- `message` 用于用户展示或调试输出。
- 诊断细节可作为 `details` 保留，但不能驱动产品级业务分支。
- 入口层只负责把同一错误翻译为自身协议格式，而不能重定义成功/失败边界。

### 7.3 状态语义一致性

- 当前架构规格已经要求区分至少三种状态语义：
  - 业务写成功；
  - 投影未收敛；
  - 同步未收敛。
- 这三类状态必须在 `FRB`、`CLI`、`HTTP`、`MCP` 四个入口中保持相同含义。
- 新增入口不得把投影失败或同步失败伪装成业务写失败。
- 入口运行时故障也必须与业务故障区分，例如：
  - HTTP 监听失败；
  - MCP 服务启动失败；
  - CLI 调试入口初始化失败。

## 8. 测试与验收设计

### 8.1 测试分层

- Rust 领域与应用服务测试：验证业务规则、读写链路、DTO 语义、错误语义。
- 入口契约测试：验证 `FRB`、`CLI`、`HTTP`、`MCP` 对应用服务结果的协议映射。
- 多入口端到端验收测试：验证同一核心用例在四个入口上得到一致的可观察语义。

### 8.2 第一阶段关键验收场景

- 默认配置下，`FRB` 可用，`HTTP`、`MCP`、`CLI` 不可用。
- 启用 `HTTP`、`MCP` 或 `CLI` 后，当前 App 会话中对应入口立即可用。
- 重启 App 后，入口启用状态仍能按配置文件保持持久化。
- 通过任一入口创建 pool、加入 pool、创建 card、更新 card 后，经查询与同步收敛后，其他入口可观察到一致结果。
- 同一业务错误在四个入口上保有一致 `error.code` 语义。
- 同一降级状态在四个入口上保有一致状态语义，而不是被不同入口各自改写。

### 8.3 完成判定

- Rust 已从“主要服务 FRB 的 API 门面”收敛为“单后端内核”。
- `FRB`、`CLI`、`HTTP`、`MCP` 都只作为薄入口访问同一应用服务层。
- 四入口共享同一批核心用例，并在结果、错误和状态语义上保持一致。
- 入口启用配置由 Rust 统一持久化，并能驱动运行时真实启停。
- Flutter 不再是唯一被优先照顾的调用方，但其主路径行为保持稳定。

## 9. 规格文档变更要求

- 本设计落地前，必须先检查 `docs/specs/` 是否已经充分表达以下正式约束：
  - Rust 后端不仅服务于 FRB，还可服务于多个入口；
  - 可选入口默认关闭、由 Rust 配置持久化管理；
  - 启用后的入口在 App 运行期间保持可用；
  - 多入口共享同一业务语义、错误语义和状态语义；
  - HTTP 当前仅限本机 loopback，但未来允许演进为可配置监听。
- 如果现有正式规格未覆盖这些约束，则 implementation plan 的首批任务必须先更新相关规格文档，再进入代码实现。
- 若 `docs/specs/` 新增、删除或重命名文档，必须同步更新 `docs/specs/DIR.md`；如目录语义变化，还需同步更新 `docs/DIR.md`。
- implementation plan 必须显式把“规格文档 CRUD 与索引一致性检查”列为前置任务，而不是附带事项。
