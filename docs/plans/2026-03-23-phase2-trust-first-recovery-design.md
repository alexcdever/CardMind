# Phase 2 信任优先恢复设计

**日期**: 2026-03-23  
**状态**: 已确认设计，待撰写实施计划  
**作者**: OpenCode

---

## 1. 背景

`docs/plans/2026-03-23-next-phase-roadmap-design.md` 已将下一阶段 roadmap 收束为三层结构：

1. `Phase 1`：证明跨设备延续成立。
2. `Phase 2`：把延续变成可解释、可恢复、可依赖。
3. `Phase 3`：谨慎评估数据流转扩展能力。

其中，`Phase 1` 的边界已经明确为：先建立“跨设备延续成立 + 最低恢复能力成立”的基线。该阶段关注的是，当轻度波动或单步阻塞出现时，用户至少不会立刻失去方向感，能够判断是不是同一份数据、内容是否安全、下一步该做什么。

但这还不足以支撑长期信任。

当异常不是一次性的轻度波动，而是持续存在、重复出现，或需要用户跨越多个状态阶段来理解系统时，用户的核心担心会从“现在下一步是什么”进一步升级为：

1. 我的内容是不是已经丢了。
2. 我的本地内容现在到底还安不安全。
3. 我是不是已经被系统踢出了原来的主路径，只能重新开始。

因此，`Phase 2` 不能简单理解为“补更多恢复流程”或“让同步更强”，而必须被定义为一轮新的产品成立判断：在异常持续存在时，CardMind 是否仍能让用户信任自己的内容安全，并继续停留在同一条主路径中。

---

## 2. 目标

本设计的目标是为 `Phase 2` 定义一个单一、稳定且可用于后续 planning 的产品目标：

> 当同步、查询收敛或设备切换出现持续性或重复性异常时，用户依然信任本地内容仍安全、没有丢失，并理解自己仍处在同一条内容延续路径上。

这意味着 `Phase 2` 的主要问题不是“如何让系统更自动恢复”，而是：

1. 如何优先建立用户对本地内容安全的稳定认知。
2. 如何避免用户把“同步尚未恢复完成”误解为“内容已经丢了”。
3. 如何让用户在异常持续存在时，仍愿意继续记录、查看、等待或执行恢复动作，而不是退出主路径。

本设计服务的是产品优先级和后续实现边界判断，而不是直接下沉为页面稿、实现细节或测试代码清单。

---

## 3. 设计原则

`Phase 2` 的设计 MUST 同时遵守以下原则：

1. **信任优先于恢复强度**：先解决“用户是否相信内容还在”，再讨论“系统能否更快恢复”。
2. **本地内容安全优先于跨设备完成态**：只要本地内容仍安全，系统就不应让用户首先感知为失败或重置。
3. **状态分层必须清晰**：必须稳定区分“本地内容安全”和“跨设备同步未完成”这两类不同事实。
4. **异常下主路径不断裂**：当本地内容仍安全时，记录、查看、继续编辑不应被同步异常整体阻断。
5. **恢复语义服务信任，不服务功能包装**：恢复动作、状态表达和自动化能力只有在增强信任时才应进入近期优先级。
6. **不把外层扩张带入 Phase 2**：设置中心补齐、协作扩展、平台化叙事都不能压过本阶段的内容安全信任目标。

---

## 4. Phase 1 与 Phase 2 的边界

为了避免 roadmap 漂移，本设计明确 `Phase 1` 与 `Phase 2` 的判定边界。

### 4.1 Phase 1 解决什么

`Phase 1` 解决的是：

1. 异常出现时，不要立刻断路。
2. 用户至少能判断当前是不是同一份数据。
3. 用户至少能判断内容是否安全。
4. 用户至少能知道下一步是什么。

它关注的是 `最低恢复能力` 是否成立，适用于轻度波动、短暂等待、单步受阻等代表性场景。

### 4.2 Phase 2 解决什么

`Phase 2` 解决的是：

1. 当异常持续存在时，用户是否仍相信内容没有丢。
2. 当异常重复出现时，用户是否仍能稳定区分“本地内容安全”与“同步尚未完成”。
3. 当系统处于恢复过程中的多个阶段时，用户是否仍认为自己在继续同一份内容，而不是被迫重新开始。

因此，`Phase 2` 不是把 `Phase 1` 的提示再做大一点，而是把“最低恢复能力”推进为“稳定的内容安全信任”。

### 4.3 不属于 Phase 2 的内容

以下内容不属于 `Phase 2` 的主目标：

1. 为了看起来更强而扩展复杂协作恢复流。
2. 为了版本包装而提前引入平台化或更大产品叙事。
3. 把设置入口补齐当作信任问题的主解决方案。
4. 只增强自动恢复编排，但无法提升内容安全信任的工程能力。

---

## 5. Phase 2 的核心问题定义

基于本次澄清，`Phase 2` 的第一优先级信任断裂点被定义为：

> 用户怀疑内容已经丢了，或不知道本地内容是否仍然安全。

这一定义会直接约束后续所有产品判断：

1. 如果某项改动首先增强的是“本地内容仍安全”的认知，它属于 `Phase 2` 候选项。
2. 如果某项改动只是在优化状态机、自动化流程或界面完整性，但不能增强上述认知，它不应优先进入 `Phase 2`。

进一步地，本阶段在异常时优先向用户承诺的第一件事应当是：

> 你的本地内容还在，先不要慌。

这一定义意味着：

1. `Phase 2` 的首要承诺不是“系统马上恢复成功”。
2. `Phase 2` 的首要承诺也不是“用户现在立刻去做一个操作”。
3. `Phase 2` 的首要承诺是先稳定建立本地内容安全感，再引导用户理解系统当前位于哪个恢复阶段。

## 6. 领域语义定义

为了让 `Phase 2` 可以直接进入 implementation planning，本节对核心术语给出可判定定义。

### 6.1 本地内容安全

`local_content_safety` 在 `Phase 2` 中统一定义为三个规范枚举：`safe` / `read_only_risk` / `unknown`。

#### `safe`

`safe` 指以下事实同时成立或可被稳定确认：

1. 用户当前设备上的最新本地内容仍可被读取。
2. 用户当前设备上的最新本地内容仍可继续写入，且不会因为同步异常被强制拒绝。
3. 系统没有证据表明该内容已经损坏、被回滚、被替换或处于不可依赖状态。

以下情况 **不等于** 本地内容不安全：

1. 另一台设备暂时还没看到最新内容。
2. 当前同步尚未完成。
3. 系统正在重试、重连或等待查询收敛完成。

进入条件：

1. 本地最新内容可读且可继续低风险写入。
2. Rust 没有收到足以升级为风险态的证据。

退出条件：

1. 出现“可读但不可安全写”的风险证据，升级到 `read_only_risk`。
2. 出现无法继续诚实承诺本地内容可依赖的证据，升级到 `unknown`。

#### `read_only_risk`

`read_only_risk` 指以下事实成立：

1. 用户仍可读取当前本地内容。
2. 系统不能再诚实承诺继续写入属于低风险主路径行为。
3. 当前问题仍未升级到“连本地内容是否可依赖都无法确认”的程度。

进入条件：

1. 内容仍可读，但新的本地写入结果存在明显风险。
2. 系统检测到本地写入路径不稳定、可能回滚、或需要先恢复后再继续写。

退出条件：

1. 本地写入重新被确认低风险，回到 `safe`。
2. 连当前本地结果是否仍可依赖都无法确认，升级到 `unknown`。

允许操作：查看、检查状态、执行明确恢复动作。  
禁止操作：把继续写入伪装成低风险正常主路径。

#### `unknown`

以下情况属于 `unknown`：

1. 系统无法确认本地最新写入是否仍可稳定读取。
2. 系统检测到本地状态可能损坏、冲突未决或持久化结果不明。
3. 系统无法继续诚实地向用户承诺“你的本地内容还在且可依赖”。

进入条件：

1. 本地内容可依赖性本身已经无法确认。
2. 风险证据互相冲突且无法被 Rust 归一化为更低一级状态。

退出条件：

1. 本地内容重新被确认至少为 `read_only_risk`。
2. 本地内容重新被确认完全安全，回到 `safe`。

允许操作：查看状态、执行明确恢复动作。  
禁止操作：继续承诺内容安全；继续开放高风险写入。

### 6.2 主路径

`主路径` 在本设计中指个人多设备笔记使用中的连续行为链：

1. 在当前设备记录或查看内容。
2. 在另一台个人设备上的 app 实例继续读取或继续记录同一份内容。
3. 在发生异常时，仍不需要把这段经历理解为重新导入、重新初始化或重新开始。

因此，凡是会把用户从“继续同一份内容”推回到“是不是要重来”的体验，都应被视为主路径断裂风险。

### 6.3 恢复阶段

`恢复阶段` 指系统为了回到稳定状态而处于的中间阶段表达。`Phase 2` 至少允许以下阶段存在，并要求它们共享一致语义：

1. `稳定态`：当前不存在活跃恢复步骤，也不存在异常降级；若某个子链路尚未完全收敛，只能表示平稳等待，而不能伪装成异常恢复中。
2. `等待中`：系统仍在推进恢复，但当前没有证据表明本地内容不安全。
3. `重试中`：系统正在重新尝试同步、查询收敛或实例连接，用户可理解为“仍在恢复同一份内容”。
4. `需用户动作`：系统无法仅靠自身恢复，需要用户执行明确动作，但本地内容安全状态必须先被说明。
5. `安全无法确认`：系统不能继续承诺本地内容仍安全，此时必须显式降级，而不能伪装成普通等待或重试。

### 6.4 查询收敛

`查询收敛` 在本设计中指：笔记数据创建或更新进入 `LoroDoc` 后，通过订阅机制或等价链路把变更同步到 `SQLite`，从而更新查询侧读模型结果的过程。

它与 `同步`、`设备切换` 的关系约束如下：

1. `同步` 关注不同 app 实例之间的数据与状态是否被成功传递。
2. `查询收敛` 关注 `LoroDoc -> SQLite` 读模型是否已经收敛到可查询结果。
3. `设备切换` 关注用户是否在另一台设备上的 app 实例继续同一份内容。

在 `Phase 2` 中，三者不是同一个概念，但都会影响用户对“内容是否仍安全、当前是否仍在同一路径上”的判断：

1. 只要本地内容仍安全，`同步失败`、`查询未收敛`、`设备切换未完成` 都不应被表达为“内容已经丢失”。
2. `查询收敛` 不等于跨设备同步，它描述的是本地 `LoroDoc -> SQLite` 读模型收敛状态。
3. `设备切换` 不等于系统链路状态，它描述的是用户在另一台设备上的 app 实例继续同一份内容时的体验场景。
4. 三者可以共享“本地内容安全 / 安全无法确认 / 是否仍在同一路径”这组高层判断，但不能混成同一个底层状态维度。

### 6.5 明确恢复动作

`明确恢复动作` 指系统能够向用户提供一个单步、可理解、不会制造新歧义的下一步，例如：

1. 重试当前同步。
2. 重新连接当前 app 实例会话。
3. 重新检查当前状态。
4. 返回原设备上的 app 实例确认最新内容。

以下内容 **不算** 明确恢复动作：

1. 只告诉用户“系统有问题”但不给下一步。
2. 同时给出多个互相竞争的建议而不说明优先级。
3. 让用户自行猜测是等待、重试还是放弃。

## 7. 架构单元与职责边界

为了保证后续 planning 能稳定切分 Rust 与 Flutter 的职责，`Phase 2` 需要以下最小架构单元。

### 7.1 Rust 语义契约层

职责：产出稳定的恢复语义真相，至少包括：

1. 本地内容是否安全。
2. 同步状态、查询收敛状态、设备切换相关连续性状态是否处于可接受范围。
3. 当前恢复阶段是什么。
4. 是否存在明确恢复动作。
5. 当前是否仍是同一条内容延续路径。

该层禁止把这些判断留给 Flutter 再推断。

输入：同步状态、查询收敛状态、设备切换相关状态、错误码、持久化结果。  
输出：稳定契约字段与允许操作边界。  
归属：Rust。  
测试方式：contract tests、integration tests、unit tests。

Rust 输入语义边界约束：

1. `同步状态`、`查询收敛状态`、`设备切换相关状态` 属于原始入口状态，必须先在 Rust 内归一化后才能对外输出。
2. `错误码` 与 `持久化结果` 属于内部证据，不应直接原样泄露给 Flutter 作为恢复判断真相。
3. 未知错误或新增错误码必须先映射到既有契约语义：`safe`、`read_only_risk`、`unknown`、`same_path`、`path_at_risk`、`path_broken` 等，而不是让 Flutter 补推断。
4. 如果 Rust 无法把某个新异常归入既有安全语义，默认必须保守提升到 `unknown`，而不是继续乐观表达为普通恢复中。

### 7.2 Flutter 呈现层

职责：只消费 Rust 已定义的稳定语义，并把它们如实呈现给用户。

该层必须遵守：

1. 不自行发明“内容安全”判断。
2. 不自行推断“其实只是同步慢一点”。
3. 不在 Rust 没有给出确定语义时擅自承诺内容安全。

输入：Rust 输出的稳定契约字段。  
输出：用户可见的状态表达、动作入口、禁用规则。  
归属：Flutter。  
测试方式：widget tests、integration tests。

### 7.3 FFI / 状态分发接口

职责：把 Rust 契约以稳定方式分发给 Flutter 消费。

最小约束：

1. Flutter 必须消费 `恢复语义快照`，而不是自己拼装多个底层状态源。
2. 该快照在页面进入、状态变化、恢复动作完成后都必须可重新获取。
3. 若后续实现选择流式更新，也必须保证任意时刻都能读取到一份完整快照，而不是只收到碎片事件。
4. 更新时序责任在 Rust：Rust 负责先完成底层证据归一化，再把稳定契约分发给 Flutter。
5. 若 Flutter 无法拿到最新快照、快照刷新失败或流式更新中断，默认必须使用 `上一次保守快照`；如果连保守快照也不可得，则收敛到 `local_content_safety = unknown` 与 `recovery_stage = unsafe_unknown`。

输入：Rust 内部归一化后的契约状态。  
输出：Flutter 可消费的完整恢复语义快照。  
归属：Rust 定义接口，Flutter 消费接口。  
测试方式：contract tests 验证快照字段完整性，integration tests 验证更新后 UI 收敛。

最小恢复语义快照必须至少包含：

1. `local_content_safety`
2. `sync_state`
3. `query_convergence_state`
4. `instance_continuity_state`
5. `recovery_stage`
6. `next_action`
7. `continuity_state`
8. `allowed_operations`
9. `forbidden_operations`

其中 1-7 是基础契约字段，8-9 是 Rust 直接输出的规则结果，不允许 Flutter 自行派生。

#### 保守快照与重启恢复规则

1. `上一次保守快照` 的真相源在 Rust，不在 Flutter。
2. Flutter 只消费 Rust 提供的最近一份完整快照，不负责持久化保守语义。
3. 应用重启后，Rust 必须优先基于持久化底层证据重算当前契约快照。
4. 只有在重算尚未完成但最近一次 Rust 快照仍可读取时，才允许短暂回退到该 `上一次保守快照`。
5. 如果既无法重算，也无法读取最近一次 Rust 快照，则必须输出 `local_content_safety = unknown` 与 `recovery_stage = unsafe_unknown`。
6. 一旦 Rust 重算得到更新结果，Flutter 必须立即丢弃旧快照并切换到新的 Rust 输出。

### 7.4 主路径可用性规则集

职责：根据 Rust 契约决定当前哪些主路径操作仍然允许。

最小约束：

1. 当本地内容仍安全时，记录、查看、继续编辑默认应保持可用。
2. 当本地内容安全无法确认时，系统必须进入明确降级，而不是继续假装主路径可完全正常使用。

该单元不是独立产品模块，而是一组规范性契约规则：其真相源在 Rust，执行呈现在 Flutter。

输入：Rust 契约字段组合。  
输出：允许操作 / 禁止操作规则。  
归属：规则源在 Rust，呈现在 Flutter。  
测试方式：contract tests 验证规则输出，widget/integration tests 验证规则呈现。

### 7.5 恢复动作规则集

职责：统一定义哪些动作属于 `等待系统恢复`，哪些动作属于 `需要用户明确介入`。

该边界必须由 Rust 契约层输出，Flutter 只负责把动作作为单步建议展示给用户。

该单元不是独立页面或服务，而是一组规范性契约规则：Rust 决定是否需要动作以及动作类型，Flutter 决定如何在当前入口把它展示成单一步骤。

输入：Rust 契约字段组合与当前入口场景。  
输出：单步恢复动作或 `none`。  
归属：规则源在 Rust，呈现在 Flutter。  
测试方式：contract tests 验证动作枚举，integration/widget tests 验证动作可达性与唯一性。

## 8. 语义组合与状态迁移约束

为了让 `Phase 2` 能直接进入 planning，本节定义契约字段之间的最小组合规则与迁移约束。

### 8.1 最小契约字段

`Phase 2` 的最小契约面包括：

1. `local_content_safety`：`safe` / `read_only_risk` / `unknown`
2. `sync_state`：`ready` / `recovering` / `blocked`
3. `query_convergence_state`：`ready` / `pending` / `blocked`
4. `instance_continuity_state`：`ready` / `recovering` / `blocked`
5. `recovery_stage`：`stable` / `waiting` / `retrying` / `needs_user_action` / `unsafe_unknown`
6. `next_action`：`none` / `retry_sync` / `retry_query_convergence` / `reconnect_instance` / `recheck_status` / `return_to_source_instance`
7. `continuity_state`：`same_path` / `path_at_risk` / `path_broken`

其中：

1. `sync_state` 负责表达不同 app 实例之间的数据同步状态。
2. `query_convergence_state` 负责表达 `LoroDoc -> SQLite` 读模型查询结果是否收敛。
3. `instance_continuity_state` 负责表达用户在另一台设备上的 app 实例继续使用同一份内容时，连续性是否处于可接受范围。
4. `recovery_stage` 负责表达当前恢复过程所处阶段，其中 `stable` 只表示当前没有活跃恢复步骤。

三类子状态统一定义如下：

1. `ready`：该子链路已达到当前可接受稳定态。
2. `recovering`：该子链路仍在推进，但尚未恢复到稳定态。
3. `blocked`：该子链路当前无法继续推进，需要显式处理。

子状态与 `recovery_stage` 的最小判定边界如下：

1. 任一子状态为 `blocked` 时，`recovery_stage` 只能是 `needs_user_action` 或 `unsafe_unknown`。
2. 至少一个子状态为 `recovering` 或 `pending`，且不存在 `blocked` 时，`recovery_stage` 可以是 `stable`、`waiting` 或 `retrying`。
3. `sync_state = ready`、`query_convergence_state = ready`、`instance_continuity_state = ready` 时，`recovery_stage` 的唯一合法值应为 `stable`。
4. 其他组合默认视为非法组合，Rust 必须在对外输出前先归一化到上述允许组合之一。

`next_action` 还必须满足：

1. 始终最多只有一个主动作。
2. `next_action = none` 表示当前不要求用户主动介入。
3. `recovery_stage = needs_user_action` 时，`next_action` 必须不是 `none`。
4. `next_action` 允许在未来扩展新枚举值，但新增值必须继续满足“单步、可理解、不制造竞争建议”这一约束。

### 8.2 合法组合规则

至少应满足以下组合约束：

1. `local_content_safety = safe` 时：
   - `continuity_state` 可以是 `same_path` 或 `path_at_risk`
   - 不能直接是 `path_broken`，除非系统已经要求用户重新开始
2. `local_content_safety = read_only_risk` 时：
    - `continuity_state` 至少应升级为 `path_at_risk`
    - `sync_state`、`query_convergence_state`、`instance_continuity_state` 中至少一个只能是 `recovering`、`pending` 或 `blocked`
    - `recovery_stage` 只能是 `retrying`、`needs_user_action` 或 `unsafe_unknown`
    - `next_action` 只能是 `none`、`retry_sync`、`retry_query_convergence`、`recheck_status` 或 `return_to_source_instance`
3. `local_content_safety = unknown` 时：
   - `recovery_stage` 必须是 `unsafe_unknown`
   - `continuity_state` 不能仍是 `same_path`
4. `recovery_stage = needs_user_action` 时：
   - `next_action` 不能是 `none`
   - `next_action` 必须保证最多一个主动作
5. `next_action = none` 时：
   - `recovery_stage` 只能是 `stable`、`waiting` 或 `retrying`
   - 不能与 `needs_user_action` 同时出现
6. 任一子状态为 `blocked` 时：
   - `recovery_stage` 只能是 `needs_user_action` 或 `unsafe_unknown`
   - 若 `next_action = none`，则只能与 `unsafe_unknown` 组合出现
7. 三个子状态都为 `ready` 时：
   - `recovery_stage` 必须是 `stable`
8. `continuity_state = path_broken` 时：
    - 系统必须已经无法再把当前经历诚实表达为“继续同一份内容”
    - Flutter 不得继续使用“只是还在恢复”的弱化表述

### 8.3 continuity_state 判定表

| 条件 | continuity_state |
|---|---|
| `local_content_safety = safe` 且系统仍可表达为恢复同一份内容 | `same_path` |
| `local_content_safety = safe` 但 `sync_state != ready`、`query_convergence_state != ready`、`instance_continuity_state != ready`，或 `recovery_stage = needs_user_action` | `path_at_risk` |
| `local_content_safety = read_only_risk` | `path_at_risk` |
| `local_content_safety = unknown` 但系统仍允许用户依赖当前本地内容做有限确认动作 | `path_at_risk` |
| `local_content_safety = unknown` 且系统必须要求用户停止依赖当前本地内容 | `path_broken` |
| 系统要求重新初始化、重新导入、重新建链路，或必须停止依赖当前本地内容 | `path_broken` |

进一步约束：

1. `path_at_risk` 表示主路径仍可被挽回，但用户已经不能被动假设“一切正常”。
2. `path_broken` 表示系统已不能再诚实表达“你正在继续同一份内容”。
3. Rust 必须输出这一判断，Flutter 不得把 `path_broken` 弱化成普通等待或重试文案。
4. “必须停止依赖当前本地内容”至少包括以下证据之一：
   - 当前本地结果可能损坏且不可验证
   - 当前本地结果可能回滚且不可确认最终版本
   - 系统必须要求用户放弃当前本地路径，转向重建或回源确认

### 8.4 状态迁移约束

最小迁移规则如下：

1. `safe -> read_only_risk`：当系统仍能读取已有内容，但不能再诚实承诺继续写入为低风险时触发。
2. `read_only_risk -> unknown`：当系统连“当前本地结果是否仍可依赖”都无法确认时触发。
3. `waiting -> retrying`：当系统已开始主动重试恢复动作时触发。
4. `retrying -> needs_user_action`：当自动恢复无法继续推进，且必须要求用户执行单步动作时触发。
5. 任意阶段 -> `unsafe_unknown`：当本地内容安全无法确认时立即触发，优先级高于普通等待与重试。
6. 任一子状态 `recovering -> ready`：当对应子链路恢复到稳定态时触发。
7. 任意 `continuity_state = path_at_risk` -> `same_path`：仅当本地内容重新被确认安全，且系统已不再要求用户把经历理解为中断后重来时触发。
8. `needs_user_action -> waiting`：当用户已完成要求动作，且系统重新进入自动恢复推进阶段时触发。
9. `unsafe_unknown -> needs_user_action`：仅当系统已重新获得最低可信证据，并能给出单步动作时触发。
10. `unsafe_unknown -> stable` 或 `retrying`：只有在 Rust 重新确认本地内容不再处于未知风险后才允许发生。

### 8.5 退出降级态约束

系统只有在以下条件同时成立时，才能退出降级态：

1. 本地内容重新被确认安全。
2. 当前恢复阶段不再是 `unsafe_unknown` 或 `needs_user_action`。
3. 用户不再需要把当前经历理解为“中断后重来”。

### 8.6 冲突证据处理规则

当底层证据互相冲突、状态瞬时抖动、或旧状态与新证据不一致时，Rust 契约层必须按以下顺序裁决：

1. `unknown` 优先于 `read_only_risk`，`read_only_risk` 优先于 `safe`。
2. `unsafe_unknown` 优先于 `needs_user_action`，`needs_user_action` 优先于 `retrying`，`retrying` 优先于 `waiting`。
3. 风险态优先于乐观恢复态；只要有充分证据表明不能继续诚实承诺安全，就不得继续输出普通等待或重试。
4. 当新证据尚不足以把高风险状态降回低风险状态时，必须保持原有更保守的输出，不得因瞬时波动立即乐观回退。
5. 未知错误、新错误码或未预期组合，默认必须收敛到更保守的既有契约语义，而不是暴露未定义状态给 Flutter。

### 8.7 规则输出归属

为了避免 Flutter 二次推断，以下内容都必须由 Rust 作为直接契约结果输出，而不是由 Flutter 派生：

1. `next_action`
2. `continuity_state`
3. 允许操作 / 禁止操作规则集

Flutter 只负责把这些结果映射为界面表达与可达交互。

### 8.8 允许操作矩阵

| 契约状态 | 允许操作 | 禁止操作 |
|---|---|---|
| `local_content_safety = safe` | 查看、继续编辑、等待、重试、执行单步恢复动作 | 把状态表达为内容已丢失 |
| `local_content_safety = read_only_risk` | 查看、检查状态、执行明确恢复动作 | 继续把写入视为低风险正常路径 |
| `local_content_safety = unknown` | 查看状态、执行明确恢复动作 | 承诺内容仍安全；继续开放高风险写入 |

该矩阵是跨场景统一规则，不得只散落在各个 UI 分支里。

## 9. 纳入 Phase 2 的能力边界

`Phase 2` 只纳入那些能够直接建立“内容还在、本地仍安全、系统没有把我踢出主路径”信任的能力。

### 9.1 应进入范围的能力

1. **更稳定的内容安全表达**  
   当异常持续时，系统必须持续回答：本地内容是否仍可读、可继续写、是否已经安全保留。

2. **更稳定的状态分层**  
   系统必须明确拆开以下两层事实：
   - 本地内容是否安全
   - 跨设备同步是否已经恢复完成

   这样用户才不会把“另一台设备还没看到最新内容”误解为“内容已经丢了”。

3. **恢复阶段可理解**  
   当系统处于等待、重试、重新连接实例、重新收敛查询结果等阶段时，用户需要理解这些阶段是在恢复同一份内容，而不是开始一套新的迁移或初始化流程。

4. **异常下仍可继续使用**  
   只要本地内容仍安全，记录、查看、继续编辑等主路径行为就不应被同步异常整体阻断。

### 9.2 不应进入范围的能力

1. 不将复杂协作恢复、多成员数据流转或平台外延纳入本阶段主线。
2. 不把设置中心补齐当作内容安全信任的替代方案。
3. 不把“自动恢复编排更复杂”直接当作价值目标。
4. 不用“功能更多”来替代“信任更强”的成立标准。

## 10. 代表性场景矩阵

为了让后续 planning 与测试规划有稳定依据，`Phase 2` 至少覆盖以下代表性场景。

### 10.1 持续同步失败，但本地仍可读写

1. 用户认知目标：知道内容没丢，只是跨设备同步还没恢复。
2. 系统表达要求：明确区分 `本地安全` 与 `同步失败`。
3. 允许操作：继续查看、继续编辑、重试同步。
4. 禁止表达：把该状态直接呈现为“内容可能已经丢失”。

### 10.2 重复重连，但本地内容仍安全

1. 用户认知目标：知道系统在恢复同一份内容，而不是重新初始化。
2. 系统表达要求：把状态表达为恢复阶段，而不是新的开始流程。
3. 允许操作：等待、重试、查看当前内容。
4. 禁止表达：让用户误以为必须重新建池、重新导入或重新开始。

### 10.3 设备切换未完成，另一台设备上的 app 实例暂未看到最新内容

1. 用户认知目标：知道本地最新内容仍安全，只是另一台设备上的 app 实例尚未恢复到完成态。
2. 系统表达要求：说明这是“另一台设备上的 app 实例尚未恢复完成”，不是 `本地内容丢失`。
3. 允许操作：继续在当前设备工作、等待恢复、检查状态。
4. 禁止表达：把另一台设备未更新等同于数据已经消失。

### 10.4 查询收敛异常，但本地内容仍可继续写

1. 用户认知目标：知道当前问题发生在 `LoroDoc -> SQLite` 读模型收敛层，不是本地写入层，也不等于跨设备同步失败。
2. 系统表达要求：明确说明本地可继续使用，但查询侧结果尚未恢复稳定。
3. 允许操作：继续本地记录、查看当前内容、执行恢复动作。
4. 禁止操作：因为查询收敛异常而整体阻断本地记录。

### 10.5 本地内容安全无法确认

1. 用户认知目标：知道系统此刻不能诚实承诺本地内容仍安全。
2. 系统表达要求：明确进入 `安全无法确认` 降级态，不伪装成普通等待或重试。
3. 允许操作：仅保留不会进一步扩大风险的最小必要动作，例如查看状态、执行明确恢复动作。
4. 禁止操作：继续向用户承诺“内容肯定没问题”；继续把高风险写入当作正常主路径行为。

### 10.6 本地内容可读，但已不可安全写入

1. 用户认知目标：知道历史内容仍在，但当前设备不再处于可放心继续写入的状态。
2. 系统表达要求：明确区分 `可读` 与 `可安全继续写`，不能把“还能看到内容”伪装成“仍可正常继续记录”。
3. 允许操作：查看已有内容、执行状态检查、执行明确恢复动作。
4. 禁止操作：继续把新的高风险写入当作正常主路径行为；继续承诺“你的本地内容完全没问题”。

### 10.7 本地冲突未决或可能发生回滚

1. 用户认知目标：知道系统无法继续把当前状态表达为稳定安全，而不是单纯“同步慢一点”。
2. 系统表达要求：明确进入 `安全无法确认`，并说明当前问题不是普通等待或重试。
3. 允许操作：查看状态、执行明确恢复动作、保守地确认现状。
4. 禁止操作：把该状态继续表达为“仍是普通恢复中”；让用户误以为可以无风险继续依赖当前写入结果。

各高风险场景下的 `next_action` 约束如下：

1. `本地内容安全无法确认`：`next_action` 只能是 `recheck_status` 或 `return_to_source_instance`。
2. `本地内容可读，但已不可安全写入`：`next_action` 只能是 `recheck_status`、`retry_sync` 或 `return_to_source_instance`。
3. `本地冲突未决或可能发生回滚`：`next_action` 只能是 `recheck_status` 或 `return_to_source_instance`，不得继续给出乐观写入相关动作。

### 10.8 应用重启后仍处于异常恢复中

1. 用户认知目标：知道异常状态不是因为页面重开就“消失”，而是系统仍在处理同一份内容的恢复。
2. 系统表达要求：应用重启或页面重新进入后，仍能恢复上一次的保守契约判断，而不是默认回到乐观正常态。
3. 允许操作：查看状态、继续执行先前明确恢复动作、在安全前提下继续本地使用。
4. 禁止操作：因为重启而丢失降级态、丢失 `next_action`，或把高风险状态重新表达成普通正常态。

该场景的恢复接口约束如下：

1. Rust 必须优先基于持久化的底层证据重新计算契约，而不是仅依赖 Flutter 缓存。
2. 若底层证据不足以重算出更低风险状态，重启后必须恢复到不低于上一次的保守语义。
3. 如果重算失败，默认进入 `local_content_safety = unknown` 与 `recovery_stage = unsafe_unknown`，而不是乐观回退到正常态。

## 11. Phase 2 的成立信号

`Phase 2` 是否成立，不应以“恢复流程变多”或“系统能力看起来更强”来判断，而应以用户是否仍然信任内容安全并留在主路径里来判断。

### 11.1 核心成立信号

1. 当同步、查询收敛或设备切换出现持续性异常时，用户不会首先怀疑“我的内容丢了”。
2. 用户能够稳定区分：
   - 本地内容仍安全
   - 跨设备状态暂未恢复完成
3. 用户不会因为状态不明，就被迫把当前体验理解为重新导入、重新开始或停止使用。
4. 用户在异常期间仍愿意继续记录、查看或等待恢复，而不是立即失去信任。

### 11.2 最小验收口径

以下条目属于进入 implementation plan 前可直接使用的最小验收口径：

1. 在 `持续同步失败，但本地仍可读写` 场景中，系统必须明确表达：
   - 本地内容仍安全
   - 跨设备恢复未完成
   - 用户仍可继续本地使用
2. 在 `设备切换未完成` 场景中，系统必须避免把另一台设备上的 app 实例未看到最新内容表达为“内容已经丢失”。
3. 在 `查询收敛异常，但本地仍可继续写` 场景中，系统必须继续允许本地记录主路径存在，并且不得把查询收敛问题误报成跨设备同步问题。
4. 在 `本地内容安全无法确认` 场景中，系统必须进入显式降级：
   - 不再承诺内容安全
   - 给出明确恢复动作或状态检查路径
   - 禁止把高风险状态伪装成普通等待态
5. 在 `本地内容可读，但已不可安全写入` 场景中，系统必须明确表达：
   - 内容仍可查看
   - 当前不应继续把写入视为低风险主路径行为
   - 用户需要等待恢复或执行明确恢复动作
6. 在 `本地冲突未决或可能回滚` 场景中，系统必须禁止继续用“同步尚未完成”掩盖风险，并显式说明当前安全状态无法确认。

planning 阶段的最小验收模板必须统一采用以下格式：

1. 给定输入证据
2. Rust 必须输出的契约字段组合
3. Flutter 必须显示的事实
4. Flutter 必须禁止的表达
5. 允许与禁止的交互

### 11.3 建议验证口径

1. 在包含持续性异常的 walkthrough 或脚本化演示中，用户能主动复述：
   - 内容还在
   - 本地仍安全
   - 只是系统还没有把另一台设备上的 app 实例恢复到完成态
2. 在连续异常场景中，用户能够区分以下三类状态，而不是把它们混为一谈：
   - 仍可继续本地使用
   - 需要等待系统恢复
   - 需要执行明确恢复动作
3. 在产品评审中，任何拟进入 `Phase 2` 的工作项都必须能回答：
   - 它是在增强内容安全信任
   - 还是只是在增加系统能力表面

### 11.4 接口级验收样例

以下样例可直接作为 implementation planning 的接口级验收基线：

1. 输入证据：本地最新内容可读可写；另一台设备上的 app 实例未看到最新内容；同步正在重试。  
   Rust 输出：`local_content_safety = safe`，`sync_state = recovering`，`query_convergence_state = ready`，`instance_continuity_state = recovering`，`recovery_stage = retrying`，`next_action = none`，`continuity_state = path_at_risk`。  
   Flutter 必须表达：内容仍安全，另一台设备上的 app 实例尚未恢复完成。  
   Flutter 禁止表达：内容已经丢失。

2. 输入证据：本地内容可读，但新的写入结果风险不明；系统要求用户先检查状态。  
   Rust 输出：`local_content_safety = read_only_risk`，`sync_state = ready`，`query_convergence_state = blocked`，`instance_continuity_state = ready`，`recovery_stage = needs_user_action`，`next_action = recheck_status`，`continuity_state = path_at_risk`。  
   Flutter 必须表达：已有内容仍可查看，但当前不应继续把写入视作低风险。  
   Flutter 禁止表达：当前仍可正常继续记录。

3. 输入证据：本地状态可能损坏或回滚，Rust 无法确认内容是否仍可依赖。  
   Rust 输出：`local_content_safety = unknown`，`recovery_stage = unsafe_unknown`，`next_action = recheck_status` 或 `return_to_source_instance`，`continuity_state = path_at_risk` 或 `path_broken`。  
   Flutter 必须表达：系统当前不能诚实承诺内容仍安全。  
   Flutter 禁止表达：只是同步稍慢。

4. 输入证据：系统要求用户重新初始化、重新导入或放弃当前本地内容。  
   Rust 输出：`continuity_state = path_broken`。  
   Flutter 必须表达：当前已不再是继续同一份内容。  
   Flutter 禁止表达：只是还在等待恢复。

5. 输入证据：Flutter 无法拿到最新恢复语义快照，且上一次保守快照也不可得。  
   Rust / 集成输出：`local_content_safety = unknown`，`recovery_stage = unsafe_unknown`。  
   Flutter 必须表达：当前无法确认内容安全，需要先检查状态。  
   Flutter 禁止表达：当前状态正常。

6. 输入证据：系统要求用户重新初始化当前链路，或必须放弃当前本地内容才能继续。  
   Rust 输出：`continuity_state = path_broken`，`next_action = return_to_source_instance` 或等价重建前确认动作。  
   Flutter 必须表达：当前不是同一路径上的继续恢复。  
   Flutter 禁止表达：只是同步稍慢或还在普通重试中。

### 11.5 契约到 UI 的最小判定映射

为了让 implementation planning 可以直接拆分 Rust 与 Flutter 任务，`Phase 2` 至少需要以下可验证映射：

| Rust 契约判断 | 必须表达给用户的事实 | 允许操作 | 禁止操作 |
|---|---|---|---|
| 本地安全 + 实例连续性未完成 | 内容还在，本地仍安全；另一台设备上的 app 实例尚未恢复完成 | 继续查看、继续编辑、等待、重试 | 把状态表达为内容已丢失 |
| 本地安全 + 查询未收敛 | 内容还在，本地仍安全；查询侧读模型仍在收敛 | 继续查看、继续编辑、重试查询收敛、检查状态 | 把状态表达为跨设备内容已丢失 |
| 本地安全 + 需用户动作 | 内容仍安全；系统恢复需要用户执行单步动作 | 查看、继续编辑、执行明确恢复动作 | 同时抛出多个无优先级动作 |
| 可读但不可安全写 | 历史内容仍可查看，但当前不应继续把写入视作低风险 | 查看、检查状态、执行恢复动作 | 把写入继续伪装成正常主路径 |
| 安全无法确认 | 系统当前不能诚实承诺内容仍安全 | 查看状态、执行明确恢复动作 | 继续承诺内容安全；继续把高风险写入视为正常路径 |
| 路径断裂 | 系统不能再诚实表达“你正在继续同一份内容” | 查看状态、执行明确恢复动作、返回原路径确认 | 继续用“只是还在恢复”弱化表达；继续开放正常主路径写入 |

Rust 侧必须输出足以区分以上判定的稳定语义；Flutter 侧必须如实映射，不得自行补充乐观结论。

### 11.6 同一路径与路径断裂的判定规则

`同一条内容延续路径` 在 `Phase 2` 中的判断规则应至少满足：

1. 只要系统仍能诚实表达“当前面对的是同一份内容的恢复过程”，且没有要求用户重新初始化、重新导入或放弃当前本地内容，就仍属于同一路径。
2. 如果系统必须要求用户把当前经历理解为重新开始、重新建链路、重新导入内容，或无法再把当前本地内容视为可依赖延续的一部分，则应判定为 `路径断裂`。
3. `另一台设备上的 app 实例暂未恢复完成`、`查询尚未收敛`、`系统仍在等待或重试` 本身不构成路径断裂；`本地内容安全无法确认且必须停止依赖当前内容` 才构成高风险断裂信号。

---

## 12. 与 Phase 3 的边界

`Phase 3` 仍应保持为“谨慎评估数据流转扩展能力”的后置阶段。

只有当 `Phase 2` 已经让异常下的内容安全信任稳定成立，才有资格进入更外层的数据流转扩展能力评估。

如果用户仍会频繁怀疑：

1. 内容是不是已经丢失。
2. 本地内容是不是仍然安全。
3. 自己是不是还处在同一条内容路径里。

那么项目就还没有完成 `Phase 2`，也不应转向更外层的能力扩张。

---

## 13. 对后续 planning 的要求

本设计确认后，后续 implementation planning 应围绕以下问题展开：

1. Rust 契约层需要至少输出七类稳定判断：`本地内容安全状态`、`同步状态`、`查询收敛状态`、`实例连续性状态`、`当前恢复阶段`、`明确恢复动作`、`同一路径判断`。
2. Flutter 交互层必须只消费并呈现这些稳定语义，而不自行发明恢复判断或乐观承诺。
3. `同步 / 查询收敛 / 设备切换` 必须分层建模：同步描述实例间数据传递，查询收敛描述 `LoroDoc -> SQLite` 收敛，设备切换描述用户在另一台设备上的 app 实例继续同一份内容；三者只共享高层信任判断，不共享同一个底层字段。
4. 哪些验证场景能够证明“用户不会误判内容已丢失”，并覆盖 `持续同步失败`、`查询未收敛`、`设备切换未完成`、`可读不可写`、`安全无法确认`、`冲突未决或可能回滚`。
5. 如何保证本地记录主路径在异常下继续可用；以及当本地内容安全无法确认时，如何进入显式降级并收紧允许操作。

### 13.1 规划约束补充

这里的字段名只是建议命名；字段语义、状态机规则、组合约束与迁移约束属于规范性要求。implementation plan 可以调整命名，但不得弱化这些判定维度，也不得打破第 8 节定义的组合与迁移约束。

implementation plan 不应直接滑向：

1. 更复杂的协作恢复编排。
2. 设置中心补齐。
3. 与信任建立无直接关系的功能扩张。

---

## 14. 结论

`Phase 2` 的单一主目标应正式定义为：

> 把 `Phase 1` 的最低恢复能力，推进为“异常持续存在时，用户依然信任本地内容仍安全”的连续体验。

因此：

1. `Phase 2` 的第一优先级不是更强同步，而是更强的内容安全信任。
2. 状态表达稳定性、恢复动作明确性、自动恢复能力，都只是为这一目标服务的手段。
3. 本阶段最重要的承诺是：即使系统还没完全恢复，用户也知道内容还在、本地仍安全，自己没有被踢出主路径。

这一定义可以作为后续 roadmap 细化、实现计划撰写与验收判断的稳定上游依据。
