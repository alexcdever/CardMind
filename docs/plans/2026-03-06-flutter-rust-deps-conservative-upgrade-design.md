# 2026-03-06 Flutter + Rust 依赖保守升级设计

## 1. 背景与目标

当前仓库同时包含 Flutter 与 Rust 依赖栈：

- Flutter 依赖由 `pubspec.yaml` / `pubspec.lock` 管理。
- Rust 依赖由 `rust/Cargo.toml` / `rust/Cargo.lock` 管理。

本次目标是在不跨主版本的前提下升级依赖，降低维护债务并尽量避免行为性回归。

## 2. 方案对比与结论

- 方案 A（推荐）：双端保守升级（仅补丁/次版本）
- 方案 B：仅点升少数关键库，其余冻结
- 方案 C：先激进升级再回退

结论：采用方案 A。原因是风险最低、改动面可控、可直接结合现有质量门禁验收。

## 3. 升级边界

- 不执行主版本升级。
- 优先通过锁文件刷新获得可兼容版本：
  - Flutter：`flutter pub upgrade`
  - Rust：`cargo update`
- 如发现某直接依赖因清单约束导致无法获得补丁/次版本更新，再做最小约束调整。
- 不主动进行功能改造；仅在编译/测试失败时进行最小兼容修复。

## 4. 执行流程

1. 基线检查：查看可升级项与当前工作区状态。
2. 执行升级：分别刷新 Flutter 与 Rust 依赖锁文件。
3. 验证质量：按影响范围执行 `flutter analyze`、`flutter test`、`cargo test`。
4. 结果整理：记录成功升级项、未升级原因与后续建议。

## 5. 风险与应对

- 风险：次版本引入 API 细微不兼容，导致编译或测试失败。
  - 应对：优先最小代码兼容；若风险不可控，精确锁回单个问题依赖。
- 风险：工具链环境差异导致某一侧命令失败。
  - 应对：分别报告失败点与重现命令，不扩大变更范围。

## 6. 验收标准

- Flutter 侧：`flutter analyze` 与 `flutter test` 通过。
- Rust 侧：`cargo test` 通过。
- 依赖升级结果可审计（体现在 lock 文件与命令输出）。
