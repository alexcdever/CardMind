# 用户旅程真相源设计

**日期**: 2026-03-20  
**状态**: 已确认设计，待撰写正式规格  
**作者**: OpenCode

---

## 1. 背景

`docs/specs/product.md` 已经定义了 CardMind 当前公开产品方向：面向多设备个人用户，以本地优先、低感知同步和可解释的数据流转支撑笔记数据的长期使用与跨设备可信流转。

但当前产品层仍缺少一层关键中间真相源：它不是“产品是什么”，也不是“页面怎么做”，而是“用户如何一路感知这个产品成立”。

在 `docs/plans/2026-03-19-cardmind-product-audit.md` 中，已经明确指出项目当前的关键缺口之一，是缺少把首次使用、开始记录、跨设备延续、异常恢复和长期信任串起来的用户旅程真相源。这导致 roadmap、交互设计和实现验收都容易跳过用户价值闭环，直接落入功能或工程任务视角。

因此，需要新增一份用户旅程层正式文档，用于把高层产品目标翻译成可稳定复用的用户阶段真相。

---

## 2. 目标

本设计的目标是定义一份新的、受 `docs/specs/product.md` 约束的旅程层下游产品规格，其主要用途是 `产品校准`，而不是设计稿说明、功能清单或测试用例集合。

这份文档应当：

1. 将 `docs/specs/product.md` 中的高层目标下沉为真实用户旅程。
2. 为后续 UI、交互、roadmap 和验收提供统一上游依据。
3. 明确什么情况下可以认为某段用户旅程成立，什么情况下说明旅程已经断裂。
4. 优先服务个人多设备主路径，而不是把数据池协作重新拉回主线。

---

## 3. 范围与边界

### 3.1 纳入范围

本次用户旅程真相源仅覆盖 `个人多设备主路径`，并聚焦以下五个阶段：

1. 首次理解
2. 开始记录
3. 跨设备延续
4. 异常与恢复
5. 长期使用与信任稳定

### 3.2 不纳入范围

本次文档不覆盖：

1. 数据池协作的完整扩展旅程
2. 页面布局、组件结构或视觉方案
3. Rust / Flutter / FRB / 存储结构等实现细节
4. 具体测试代码或自动化脚本设计
5. 超出当前产品阶段的未来扩展设想

### 3.3 数据池协作的定位

数据池协作不应成为本文件的主阶段之一。它只作为个人多设备主路径之外的数据流转扩展情境被引用，主要出现在：

- 跨设备延续阶段的扩展说明
- 异常与恢复阶段的扩展说明

这样可以保证用户旅程真相源始终与 `docs/specs/product.md` 中“个人多设备优先、协作扩展次之”的产品边界保持一致。

---

## 4. 文档定位与上下位关系

建议新增正式规格文档：

- `docs/specs/user-journeys.md`

该文档属于受 `docs/specs/product.md` 约束的下游产品规格，职责是承接高层产品目标并把它翻译成用户阶段真相：

- `docs/specs/product.md` 回答：`CardMind 是什么产品`
- `docs/specs/user-journeys.md` 回答：`这个产品如何在真实用户身上成立`

下游关系建议明确为：

- `docs/specs/ui-interaction.md`：将旅程承诺转化为可见交互与状态反馈
- `docs/specs/card-note.md`：支撑个人记录主路径的领域行为
- `docs/specs/pool.md`：支撑数据流转扩展能力的领域行为

因此，`user-journeys.md` 应被定义为受 `product.md` 约束的旅程层规格，用于继续约束交互、领域规格和后续验收判断。

## 5. 结构设计

建议 `docs/specs/user-journeys.md` 使用以下结构：

1. 文档导读
2. 目的与范围
3. 与 `docs/specs/product.md` 的关系
4. 用户旅程总原则
5. 阶段一：首次理解
6. 阶段二：开始记录
7. 阶段三：跨设备延续
8. 阶段四：异常与恢复
9. 阶段五：长期使用与信任稳定
10. 与下游规格和后续规划的关系

这份结构强调“先原则、后阶段、再衔接”，避免一开始就掉进局部页面或局部功能说明。

---

## 6. 每个阶段的统一记录格式

每个用户阶段 MUST 使用统一的记录格式，只包含以下四类内容：

1. **用户此时想完成什么**  
   关注用户目标，而不是系统动作。

2. **用户此时最在意什么**  
   关注用户此刻真正担心的问题，例如是否值得理解、是否能继续写、是否会丢数据、是否值得长期信任。

3. **产品必须提供的体验承诺**  
   这是该文档的核心，用来定义该阶段产品必须成立的体验结果，而不是实现方案。

4. **旅程成立信号 / 旅程断裂信号**  
   用于判断这一阶段是否真正成立，并为后续 roadmap、交互设计和验收提供依据。

每个阶段不应写入：

- 页面布局说明
- 组件级实现细节
- API / FFI / 存储约束
- 尚未确认的未来扩展能力
- 不能帮助判断旅程是否成立的杂项说明

文档级约束建议明确写入：

> 如果一条内容不能帮助判断某段用户旅程是否成立，则不应进入本文件。

## 7. 五个阶段的目标轮廓

### 7.1 阶段一：首次理解

- 用户目标：快速判断这是不是适合自己的工具
- 用户最在意：这是什么、和普通笔记有什么区别、是否必须先理解复杂同步机制
- 产品承诺重点：让用户先知道“可以先记，再逐步理解同步与流转”
- 旅程断裂信号：用户虽然能看到页面，但说不清这产品为什么存在

### 7.2 阶段二：开始记录

- 用户目标：顺利创建、编辑、查找第一批笔记
- 用户最在意：是否能立刻开始用、内容是否先归自己控制、基本读写是否稳定
- 产品承诺重点：记录动作必须低阻力，不应要求用户先理解复杂协作结构
- 旅程断裂信号：用户还没获得“记下来”的正反馈，就被空页面、导航或概念负担打断

### 7.3 阶段三：跨设备延续

- 用户目标：在另一台设备继续看到并使用同一份笔记数据
- 用户最在意：是不是同一份内容、延续是否自然、是否需要主动管理大量同步步骤
- 产品承诺重点：跨设备体验必须被感知为“连续使用”，而不是一次新的迁移或导入
- 旅程断裂信号：用户把跨设备延续感知成重新配置系统，而不是继续使用产品

### 7.4 阶段四：异常与恢复

- 用户目标：在同步延迟、失败或状态不明时仍知道该怎么办，并相信内容没白写
- 用户最在意：内容是否丢失、当前状态是什么、是否应停止使用、下一步是什么
- 产品承诺重点：异常必须可解释、可恢复，并尽量不阻断记录主路径
- 旅程断裂信号：用户一旦看到异常，就怀疑内容安全或不敢继续记录

### 7.5 阶段五：长期使用与信任稳定

- 用户目标：把 CardMind 当作长期依赖的跨设备笔记流转工具
- 用户最在意：是否能持续依赖、偶发异常后是否仍值得信任、使用成本是否逐渐降低
- 产品承诺重点：长期信任来自持续稳定的体验，而不只是偶尔成功一次同步
- 旅程断裂信号：产品功能上可用，但用户心理上仍不敢依赖

---

## 8. 用户旅程总原则

建议在正式规格中先声明若干总原则，用于约束后续所有阶段：

1. 先记录，再理解复杂机制。
2. 主路径优先于扩展能力。
3. 异常恢复属于主路径的一部分，而不是边缘补丁。
4. 可信任比功能堆叠更重要。
5. 如果某项设计不能增强个人多设备笔记流转主路径，应降低优先级。

这些原则能够让后续 roadmap、交互设计和实现验收都优先服务“旅程成立”，而不是先服务功能扩张。

---

## 9. 对后续规划的价值

这份文档完成后，应服务以下更上游的判断：

- 哪些用户阶段属于当前产品必须优先成立的主路径
- 哪些体验承诺必须先被稳定建立，哪些可以延后
- 交互、领域规格和验收在校准优先级时应参照什么用户阶段真相
- AI 在面对多个候选任务时，如何判断哪个更贴近当前产品阶段目标

换句话说，它不是替代 roadmap，而是为 roadmap、交互设计和后续规格提供更稳的判定依据。
