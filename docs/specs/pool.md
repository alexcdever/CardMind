## 1. 目的与范围

本规格定义 CardMind 数据池（Pool）领域的行为契约，用于指导任意技术实现并作为验收依据。

若实现或验收涉及项目级架构边界，本规格 MUST 与 `docs/specs/architecture.md` 一并遵守。

本规格覆盖：

1. 数据池生命周期行为。
2. 角色与权限边界。
3. 数据池元数据与不变量。
4. 池内卡片笔记引用与同步语义。
5. 错误语义、恢复动作与黑盒验收。

本规格不覆盖：

1. 编程语言、框架、存储引擎、网络协议选型。
2. 具体类名、函数签名、目录结构。
3. 非本领域行为约束所必需的底层同步实现细节。

## 2. 术语与角色

### 2.1 术语

1. Pool：协作容器，承载成员关系、入池请求与池级元数据。
2. Join Request：请求加入指定 Pool 的待处理记录。
3. Note Reference：Pool 元数据中对卡片笔记的引用（以 `noteId` 集合或等价形式表示）。
4. Dissolved：Pool 终态标记，表示该池已解散且不再接受新增协作活动。

### 2.2 角色

1. admin：池管理角色。
2. member：普通成员角色。

## 3. 规范性规则

本节采用以下术语强度：

1. MUST：必须满足。
2. FORBIDDEN：禁止。
3. SHOULD：建议满足，若不满足需有明确理由。
4. MAY：可选能力。

## 4. 生命周期与权限

### 4.1 生命周期

系统 MUST 支持以下动作：

1. 创建池。
2. 提交加入申请。
3. 审批加入申请。
4. 拒绝加入申请。
5. 成员退出池。
6. 解散池。

分域规格 SHOULD 为关键生命周期动作定义：前置条件、成功结果、失败结果、可恢复路径。

### 4.2 角色权限

1. 创建池的一方 MUST 自动成为该池首个 admin。
2. admin MUST 可执行：池信息管理、申请审批/拒绝、解散池。
3. member MUST 可执行：退出池。
4. member 执行 admin 专属动作为 FORBIDDEN。
5. 越权操作 MUST 返回稳定权限错误语义。

### 4.3 管理员不变量

1. 未解散池 MUST 至少有 1 个 admin。
2. 任何会导致未解散池 `admin=0` 的操作 MUST 被拒绝。

## 5. 领域对象与不变量

### 5.1 核心对象

1. Pool：包含池标识、池名称、解散状态等基础属性。
2. Member：包含成员标识、角色、加入状态等成员属性。
3. Join Request：包含申请标识、申请人标识、申请状态等申请属性。
4. Note Reference Set：Pool 元数据中的 `noteId` 引用集合。

### 5.2 不变量

1. 同一 Pool 内，同一 Join Request 标识 MUST 唯一。
2. 已解散 Pool 接受新的加入申请为 FORBIDDEN。
3. 已解散 Pool 新增成员为 FORBIDDEN。
4. Note Reference Set 中每个 `noteId` MUST 可被追溯到一次合法创建动作。

## 6. 池内笔记与同步语义

### 6.1 池与笔记关系

1. Pool 元数据 MUST 维护池内卡片笔记 `noteId` 引用集合。
2. 仅“创建卡片笔记”动作 MUST 触发 Pool 元数据新增 `noteId` 引用。
3. 卡片笔记后续修改与删除在 Pool 侧 MUST 视为“引用对象状态变化”，而非 Pool 元数据新增/删除语义。

### 6.2 成员间同步

1. 任一成员对池内卡片笔记的增删改，系统 MUST 最终同步到其他成员可见状态。
2. 系统 MAY 存在短暂状态延迟，但 MUST 最终一致。

### 6.3 并发一致性要求

1. 多成员并发修改同一池内笔记后，系统 MUST 最终向所有成员呈现一致的可观察结果。
2. 同一业务动作的重复提交 MUST 满足幂等预期。
3. 并发结果 SHOULD 可被业务审计与追溯。

## 7. 错误语义与恢复

1. 领域错误 MUST 使用稳定错误码语义。
2. 用户提示文案 MAY 演进，但不得改变错误码语义。
3. 可恢复失败 MUST 提供至少一个明确恢复动作（例如重试）。
4. 错误语义 MUST 独立于底层异常文本。

## 8. 验收标准（黑盒）

验收用例 SHOULD 使用 Given/When/Then 表达。

### 8.1 最小验收集

1. Given 无池上下文，When 创建池，Then 创建方成为 admin 且池处于未解散可运营状态。
2. Given 存在目标池，When 提交加入申请，Then 申请进入待处理状态。
3. Given 存在待处理申请，When admin 审批通过，Then 申请转为通过且申请人成为 member。
4. Given 存在待处理申请，When admin 拒绝，Then 申请不再待处理且结果可审计。
5. Given member 在池内，When member 退出，Then 该成员关系被移除。
6. Given 池内仅剩 1 个 admin，When 执行会导致 `admin=0` 的操作，Then 操作被拒绝并返回稳定错误语义。
7. Given Pool 已解散，When 提交加入申请或新增成员，Then 系统拒绝该操作。
8. Given 池内笔记被创建，When 观察 Pool 元数据，Then 可见新增 `noteId` 引用。
9. Given 池内成员修改或删除既有笔记，When 观察 Pool 元数据，Then 不新增新的 `noteId` 引用。
10. Given 任一成员修改池内笔记内容，When 同步收敛完成，Then 其他成员可见一致结果。

### 8.2 判定原则

1. 验收 MUST 以外部可观察行为为准。
2. 验收依赖具体实现细节（函数名、表结构、日志格式）为 FORBIDDEN。

## 9. 兼容与演进

1. 新版本规格 SHOULD 保持历史数据可解释。
2. 若实现行为变更影响本规格定义的外部行为，MUST 先更新规格后更新实现。
3. 本规格后续可扩展文件同步覆盖范围，但新增范围 MUST 显式版本化记录。
