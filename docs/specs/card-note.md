## 1. 目的与范围

本规格定义 CardMind 卡片笔记（card-note）领域的行为契约。

本规格覆盖：

1. 卡片笔记生命周期（创建、编辑、软删除、恢复）。
2. 默认列表可见性、搜索、筛选与排序规则。
3. 与数据池元数据的关联规则。
4. 协作场景下的最终一致可见结果。

本规格不覆盖：

1. 数据结构、索引结构或搜索实现方案。
2. 底层同步与合并算法细节。
3. UI 布局与组件实现细节。

## 2. 术语

1. Card Note：可被创建、编辑、软删除、恢复的笔记实体。
2. Soft Delete：逻辑删除状态，实体仍可被系统识别与处理。
3. Active Note：未软删除、默认可见的笔记。
4. Note Reference：数据池元数据中对笔记 `noteId` 的引用记录。

## 3. 规范性规则

本节采用以下术语强度：

1. MUST：必须满足。
2. MUST NOT：禁止。
3. SHOULD：建议满足，若不满足需有明确理由。
4. MAY：可选能力。

## 4. 生命周期与可见性

### 4.1 生命周期

系统 MUST 支持以下动作：

1. 创建笔记。
2. 编辑笔记。
3. 软删除笔记。
4. 恢复笔记。

### 4.2 默认可见性

1. 默认列表 MUST 仅显示 Active Note。
2. 当前版本 MAY 不提供用户侧恢复入口。

## 5. 搜索、筛选与排序

### 5.1 搜索

1. 搜索范围 MUST 仅覆盖 Active Note。
2. 搜索规则 MUST 同时匹配标题与正文。
3. 搜索匹配 MUST 为子串匹配且不区分大小写。

### 5.2 筛选

1. 系统 MUST 提供所属数据池筛选（含“全部池”语义）。
2. 系统 MUST NOT 向用户提供删除状态筛选操作。

### 5.3 排序

1. 系统 MUST 支持按更新时间排序。
2. 系统 MUST 支持按创建时间排序。
3. 默认排序 MUST 为更新时间倒序。
4. 当主排序字段相同，系统 SHOULD 保持稳定次序。

## 6. 数据池关联与协作一致性

### 6.1 池关联

1. 创建或加入数据池时，系统 MUST 自动将该用户已有笔记 `noteId` 写入池元数据。
2. 自动挂接 MUST 包含已软删除笔记。
3. 仅创建笔记会新增池侧 `noteId` 引用。
4. 对已挂接笔记执行编辑、软删除、恢复时，系统 MUST NOT 新增重复 `noteId` 引用。

### 6.2 协作一致性

1. 池内成员对笔记执行创建、编辑、软删除、恢复后，结果 MUST 最终对其他成员可见。
2. 系统 MAY 存在短暂可见延迟，但 MUST 最终一致。

## 7. 错误语义与恢复

1. 越权操作 MUST 返回稳定权限错误语义。
2. 错误语义 MUST 独立于底层异常文本。
3. 可恢复失败 MUST 提供明确恢复动作。

## 8. 验收标准（黑盒）

验收用例 SHOULD 使用 Given/When/Then 表达。

### 8.1 最小验收集

1. Given 无笔记，When 创建笔记，Then 默认列表可见该笔记。
2. Given 存在笔记，When 编辑笔记，Then 列表与详情呈现更新内容。
3. Given 存在笔记，When 软删除，Then 默认列表不可见该笔记。
4. Given 默认列表，When 执行搜索，Then 仅在 Active Note 中按标题+正文子串不区分大小写匹配。
5. Given 笔记分布在多个池，When 按池筛选，Then 结果仅包含目标池笔记。
6. Given 创建或加入数据池，When 自动挂接完成，Then 池元数据包含已有笔记 `noteId`（含已软删除笔记）。
7. Given 已挂接笔记，When 编辑/软删除/恢复，Then 池元数据不出现重复 `noteId` 引用。
8. Given 池内成员修改笔记，When 同步收敛完成，Then 其他成员可见一致结果。

### 8.2 判定原则

1. 验收 MUST 以外部可观察行为为准。
2. 验收 MUST NOT 依赖函数名、表结构、日志文本等实现细节。

## 9. 兼容与演进

1. 新版本规格 SHOULD 保持历史笔记状态可解释。
2. 若行为变更影响外部可观察结果，MUST 先更新规格后更新实现。
3. 用户侧恢复入口若未来开放，MUST 在本规格中显式新增对应行为与验收条款。
