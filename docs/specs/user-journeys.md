## 1. 目的与范围

本规格是受 `docs/specs/product.md` 约束的旅程层下游产品规格，主要用途是产品校准，而不是页面设计、实现细节或测试脚本说明。

本规格只覆盖个人多设备主路径，聚焦真实用户从首次理解到长期信任稳定的连续经历。

本规格覆盖：

1. 个人多设备主路径的五个核心阶段。
2. 每个阶段的用户目标、关注点、体验承诺与成立/断裂判断。
3. 用于约束下游交互规格、领域规格与后续规划的旅程层判断依据。

本规格不覆盖：

1. 不覆盖完整数据池协作旅程。
2. 不覆盖 UI 布局、页面结构、组件设计或视觉方案。
3. 不覆盖 Rust、Flutter、FRB、API、存储结构或其他实现细节。
4. 不覆盖具体测试代码、自动化脚本或工程任务编排。

## 2. 与 `docs/specs/product.md` 的关系

`docs/specs/product.md` 是 CardMind 当前产品方向的上位产品真相源，定义产品目标、阶段目标、目标用户与能力边界。

本规格是受其约束的旅程层下游规格，不改写产品目标，只回答这些目标如何在真实用户的连续使用过程中成立。

如果 `docs/specs/product.md` 回答“CardMind 是什么产品”，那么本规格回答“这个产品如何在真实用户身上成立”。

当本规格与更下游文档发生冲突时，下游文档 MUST 先同时满足 `docs/specs/product.md` 的产品约束与本规格的旅程约束。

## 3. 用户旅程总原则

本规格中的所有阶段判断，MUST 同时受以下总原则约束：

1. 先记录，再理解复杂机制。
2. 主路径优先于扩展能力。
3. 异常恢复属于主路径的一部分。
4. 可信任比功能堆叠更重要。

如果某项内容不能帮助判断产品是否在真实用户旅程中成立，则不应进入本规格。

## 4. 阶段一：首次理解

### 用户此时想完成什么

用户想快速判断 CardMind 是否值得开始使用，并理解它与普通云笔记的关键差异。

### 用户此时最在意什么

用户最在意这是什么产品、自己是否必须先理解复杂同步与流转机制、以及开始使用前是否要承担额外学习成本。

### 产品必须提供的体验承诺

产品 MUST 让用户在首次接触时快速理解：这不是普通云笔记，但可以先记，再逐步理解同步与流转，不需要先掌握复杂机制才能开始使用。

### 旅程成立信号

用户能够用稳定语言复述：这是一个面向个人多设备使用的笔记产品，自己可以先开始记录，再逐步理解后续的数据延续与流转能力。

### 旅程断裂信号

用户虽然接触到了产品，但仍无法判断它为什么存在，或者误以为必须先理解复杂机制后才能开始记录。

## 5. 阶段二：开始记录

### 用户此时想完成什么

用户想以低阻力开始创建、编辑和延续自己的第一批笔记内容。

### 用户此时最在意什么

用户最在意是否能直接开始记录、内容是否先归自己掌控，以及记录过程是否稳定、连续、不会被额外概念打断。

### 产品必须提供的体验承诺

产品 MUST 先给出低阻力、直接的个人记录正反馈，让用户先获得“我已经把内容记下来了”的确认，而不是要求用户先理解扩展能力或复杂结构。

### 旅程成立信号

用户在短时间内完成首次记录，并形成明确感受：记录这件事直接、稳定、可继续，产品首先服务自己的笔记主路径。

### 旅程断裂信号

用户还没有获得清晰的个人记录正反馈，就被概念负担、状态不明或额外路径切换打断，导致无法建立继续使用意愿。

## 6. 阶段三：跨设备延续

### 用户此时想完成什么

用户想在另一台个人设备上继续看到、读取并延续同一份笔记数据。

### 用户此时最在意什么

用户最在意另一台设备上的内容是不是同一份数据、延续是否自然，以及自己是否需要主动管理大量额外步骤才能继续使用。

### 产品必须提供的体验承诺

产品 MUST 让跨设备体验被感知为同一份笔记数据的连续使用，而不是一次新的迁移、导入或重新开始。

### 旅程成立信号

用户在切换设备后，仍把自己的笔记经历理解为同一条连续主路径，并自然继续使用既有内容。

### 旅程断裂信号

用户把跨设备延续感知为重新配置、重新导入或重新建立内容关系，因而不再相信自己是在继续使用同一份笔记数据。

## 7. 阶段四：异常与恢复

### 用户此时想完成什么

用户想在状态异常、延续受阻或结果不明确时，判断内容是否安全、当前状态是什么、下一步应该做什么。

### 用户此时最在意什么

用户最在意内容是否还安全、当前问题是否会影响继续记录，以及自己是否能在不失去信心的前提下恢复主路径。

### 产品必须提供的体验承诺

产品 MUST 让用户在异常发生时知道内容是否安全、当前状态是什么、下一步是什么，并相信恢复过程不会把自己排除在主路径之外。

### 旅程成立信号

用户遇到异常后，仍能判断内容安全边界、理解当前状态并采取下一步行动，且愿意继续记录或继续延续既有内容。

### 旅程断裂信号

用户一旦遇到异常，就无法判断内容是否安全、当前状态是什么或下一步该做什么，因此停止使用或不敢继续记录。

## 8. 阶段五：长期使用与信任稳定

### 用户此时想完成什么

用户想把 CardMind 作为可长期依赖的个人多设备笔记工具持续使用，而不是只在偶尔成功时短暂尝试。

### 用户此时最在意什么

用户最在意这种体验是否值得长期依赖、异常之后是否仍然可信，以及自己是否能够在持续使用中保持稳定预期。

### 产品必须提供的体验承诺

产品 MUST 通过持续稳定的记录、延续与恢复体验建立长期信任，让用户相信这是一套可以长期依赖的笔记数据延续机制，而非偶尔成功一次。

### 旅程成立信号

用户在重复使用和经历正常波动后，仍持续把 CardMind 视为可信赖的个人多设备笔记主路径，并愿意把重要内容长期放在这里延续。

### 旅程断裂信号

产品即使偶尔可用，用户仍不愿长期依赖，或在经历一次异常后就失去稳定信任，不再把它视为可靠的长期工具。

## 9. 与下游规格和后续规划的关系

本规格为下游交互规格、领域规格与后续规划提供校准依据，用于判断后续工作是否仍然服务个人多设备主路径的成立。

下游关系如下：

1. `docs/specs/ui-interaction.md` 承接本规格中的体验承诺与状态理解要求，把旅程约束转化为可感知的交互语义。
2. `docs/specs/card-note.md` 承接本规格中的个人记录主路径，细化笔记领域行为应如何支撑旅程成立。
3. `docs/specs/pool.md` 承接本规格中的数据流转扩展边界，细化扩展能力如何在不压过主路径的前提下服务旅程。

本规格约束后续规划应围绕旅程是否成立进行判断，但不把具体 roadmap 决策写死。
