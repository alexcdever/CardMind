## 1. 目的与范围

本规格定义 CardMind 的产品定位、阶段目标与能力边界，作为产品层正式真相源。

本规格覆盖：

1. 产品终极目标与当前阶段目标。
2. 当前主目标用户与核心价值承诺。
3. 核心能力、能力边界与当前非目标。
4. 与现有正式规格之间的上下位关系。
5. 产品目标变更时的验收与演进规则。

本规格不覆盖：

1. 具体技术实现、API、存储结构与目录设计。
2. 营销文案、品牌包装或未进入正式约束的概念探索。
3. 分域行为细节；这些细节由下游正式规格继续定义。

## 2. 终极目标

基于当前审计结论，CardMind 的公开产品终极目标定义为：成为一款面向多设备个人用户、用于同步和流转笔记数据的应用。

该表述是当前仓库内正式化的产品方向收束结果；若后续采用等价措辞，MUST 保持“多设备个人用户”与“同步和流转笔记数据”这两个核心含义不变。

## 3. 当前阶段目标

当前阶段优先证明以下三件事：

1. 个人用户可在本地优先前提下稳定创建、编辑、查询笔记。
2. 同一用户跨设备同步笔记数据的体验可解释、可恢复、可长期信任。
3. 数据池协作能力作为同步与流转模型的扩展能力存在，但不得压过个人多设备主路径的优先级。

## 4. 目标用户

当前主目标用户 MUST 定义为拥有多设备场景的个人用户，而不是以团队管理者、组织管理员或复杂协作流程维护者为首要对象。

这里的多设备场景包括但不限于用户在手机、桌面端或其他个人设备之间持续记录、查看与延续同一批笔记数据的需求。

## 5. 核心价值承诺

CardMind 当前对目标用户的核心价值承诺如下：

1. 本地优先：用户首先获得可直接写入与读取的个人笔记体验，不以在线状态作为记录前提。
2. 低感知同步：同一用户跨设备同步时，系统 SHOULD 尽量减少额外理解成本，让用户主要关注内容延续而非同步机制本身。
3. 数据流转：笔记数据不仅要能在单设备内存在，还要能在用户的多设备上下文以及必要的协作上下文中持续流转。
4. 可解释与可恢复：当同步或协作出现异常时，系统 MUST 让用户理解当前状态，并提供恢复路径，从而建立长期信任。

这些承诺服务的核心不是“展示复杂技术能力”，而是帮助多设备个人用户更稳定地保有、延续和流转自己的笔记数据。

## 6. 核心能力与能力边界

当前产品核心能力 SHOULD 收束为以下三类：

1. 个人笔记的创建、编辑、查询与持续使用。
2. 围绕本地优先体验展开的跨设备低感知同步。
3. 围绕笔记数据流转展开的数据池协作扩展能力。

能力边界如下：

1. 数据池协作是服务个人多设备主路径的数据流转扩展能力，不是更高优先级的独立主目标。
2. 任何新增能力若会削弱“个人用户先稳定记录，再可信同步和流转”的主路径，SHOULD 视为偏离当前阶段目标。
3. 下游规格可以细化卡片、数据池、交互与架构约束，但不得把产品主目标改写为以协作机制、平台扩张或技术展示为中心。

## 7. 非目标

当前阶段的非目标包括：

1. 不以复杂团队协作作为第一优先级。
2. 不以多角色组织流程、组织治理或企业级审批链路作为第一优先级。
3. 不以扩张型平台化能力、开放生态或大而全功能覆盖作为第一优先级。

若某项工作主要服务上述方向，而不能直接加强多设备场景下的个人笔记同步和流转主路径，则 SHOULD 降低优先级。

## 8. 与现有正式规格的关系

本规格是产品层上位真相源；现有正式规格继续承担分层细化职责：

1. `docs/specs/architecture.md` 定义项目级架构约束，回答系统如何在技术边界上支撑该产品方向。
2. `docs/specs/shared-domain-contract.md` 定义跨领域共享术语、一致性、权限与错误语义，回答下游规格在共同约束层面如何保持与本产品目标一致。
3. `docs/specs/card-note.md` 定义卡片笔记领域行为，回答个人笔记主路径需要满足哪些可观察契约。
4. `docs/specs/pool.md` 定义数据池领域行为，回答数据流转与协作扩展能力的边界与验收。
5. `docs/specs/ui-interaction.md` 定义 UI 与交互约束，回答产品承诺如何转化为可见、可恢复、可理解的用户体验。

当下游规格与本规格发生冲突时，MUST 以本规格的产品目标、阶段目标与非目标约束为准，并据此更新冲突文档。

## 9. 验收与演进规则

1. 新的公开文档、下游规格与实现说明，MUST 不违背本规格定义的终极目标、当前阶段目标、目标用户与非目标。
2. 若产品目标发生变化，MUST 先更新本产品级真相源，再更新 `docs/specs/architecture.md`、`docs/specs/pool.md`、`docs/specs/card-note.md`、`docs/specs/ui-interaction.md` 或相关实现。
3. 若只是措辞调整而非方向调整，MUST 保持本规格中的核心语义不变，尤其不得改变“多设备个人用户”与“同步和流转笔记数据”的含义。
4. 任何超出当前阶段目标的新能力，SHOULD 先通过计划文档论证，再决定是否进入本规格与下游规格。
