# <领域名称>规格

## 0. 使用说明（给 AI/实现者）

1. 目标：仅提取可观察行为约束，用于实现与验收。
2. 必读章节：`4`、`5`、`6`、`8`。
3. 禁止事项：
   - 不要把本文件当作技术实现方案。
   - 不要引入语言、框架、存储引擎或协议细节。
4. 术语强度：`MUST` > `SHOULD` > `MAY`。
5. 渐进加载顺序：
   - 首轮：`1`、`4`、`5`
   - 二轮：`6`、`7`
   - 测试轮：`8`、`9`

本规格定义<领域名称>的可观察行为约束，用于指导实现与验收，不包含技术实现细节。

## 3. 规范性规则

本节采用以下术语强度：

1. MUST：必须满足。
2. MUST NOT：禁止。
3. SHOULD：建议满足，若不满足需有明确理由。
4. MAY：可选能力。

## 4. 生命周期与权限

### 4.1 生命周期

系统 MUST 支持以下动作：

1. <动作 A>
2. <动作 B>
3. <动作 C>

每个动作 SHOULD 定义：前置条件、成功结果、失败结果、可恢复路径。

### 4.2 角色权限

1. <角色 A> MUST 可执行：<操作集合>
2. <角色 B> MUST NOT 执行：<禁止操作集合>
3. 越权操作 MUST 返回稳定权限错误语义。

## 7. 错误语义与恢复

1. 领域错误 MUST 使用稳定错误码语义。
2. 用户提示文案 MAY 演进，但不得改变错误码语义。
3. 可恢复失败 MUST 提供至少一个明确恢复动作。

## 8. 验收标准（黑盒）

验收用例 SHOULD 使用 Given/When/Then 表达。

### 8.1 最小验收集

1. Given <前置>, When <动作>, Then <可观察结果>
2. Given <前置>, When <动作>, Then <可观察结果>
3. Given <前置>, When <失败动作>, Then <错误语义 + 恢复动作>

### 8.2 判定原则

1. 验收 MUST 以外部可观察行为为准。
2. 验收 MUST NOT 依赖具体实现细节。

## 9. 兼容与演进

1. 新版本规格 SHOULD 保持历史数据可解释。
2. 若实现行为变更影响本规格定义的外部行为，MUST 先更新规格后更新实现。
3. 新增范围 MUST 显式记录版本边界。
