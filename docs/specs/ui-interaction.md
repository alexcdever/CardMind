# UI 设计与交互规格

## 文档导读（AI/渐进式加载）

### 0. 使用说明（给 AI/实现者）

1. 目标：定义 UI 的可观察交互行为与视觉边界，用于实现与验收。
2. 必读章节：`4`、`5`、`6`、`7`。
3. 禁止事项：
   - 不要把本文件当作组件实现文档。
   - 不要在本文件中锁定框架、库或样式工程结构。
4. 术语强度：`MUST` > `SHOULD` > `MAY`。
5. 渐进加载顺序：
   - 首轮：`1`、`4`
   - 二轮：`5`、`6`
   - 测试轮：`7`、`8`

### 0.1 一句话摘要

本规格定义跨端 UI 的交互强约束与视觉中约束，确保关键流程可预测、可恢复、可验收，不包含实现细节。

---

## 1. 目的与范围

本规格定义 CardMind 的 UI 设计与交互行为契约。

本规格覆盖：

1. 导航与返回语义。
2. 关键动作确认与反馈时机。
3. 空态、加载态、错误态、禁用态的可观察规则。
4. 跨端双套 UI 的布局模型与输入交互映射。
5. 关键流程（首屏入口、卡片、池、设置、同步异常）的状态与恢复语义。
6. 视觉系统的中约束边界（色彩、层级、间距、动效）。

本规格不覆盖：

1. 具体组件实现与样式文件组织。
2. 像素级与色值级冻结。

## 2. 术语

1. 关键流程：创建池、加入池、卡片创建/编辑/删除、同步失败恢复、退出确认等高频主路径。
2. 危险动作：可能导致不可逆或高影响结果的动作。
3. 可恢复失败：失败后可通过明确动作继续推进的状态。
4. 跨端一致：移动端与桌面端可布局与控件不同，但关键结果语义一致。
5. 双段返回：在主页内，非卡片页返回先回到卡片页；卡片页返回触发退出确认。
6. 双套 UI：移动端 UI 与桌面端 UI 为两套独立实现，在目标平台构建与运行时选择对应 UI，不依赖断点切换。

## 3. 规范性规则

本节采用以下术语强度：

1. MUST：必须满足。
2. FORBIDDEN：禁止。
3. SHOULD：建议满足，若不满足需有明确理由。
4. MAY：可选能力。

## 4. 交互强约束

### 4.1 导航与返回

1. 系统返回、导航栏返回、页面内返回在同一场景 MUST 保持一致语义。
2. 关键流程中的返回行为 MUST 可预测且可恢复。
3. 返回动作造成未提示的数据丢失为 FORBIDDEN。

### 4.2 危险动作确认

1. 删除、退出、解散等危险动作 MUST 提供明确确认步骤。
2. 确认信息 MUST 说明结果与影响范围。

### 4.3 反馈时机

1. 关键动作触发后 MUST 立即给出状态反馈（处理中/成功/失败）。
2. 失败反馈 MUST 包含明确下一步动作（重试/取消/返回）。

### 4.4 空态与错误态

1. 空态 MUST 说明为空原因与下一步可执行动作。
2. 不同错误语义（网络、权限、输入）MUST 可区分呈现。

### 4.5 跨端一致性

1. 移动端与桌面端 MUST 作为两套独立 UI 维护与演进。
2. 同一业务动作在不同端 MUST 产生一致的结果语义与恢复路径。
3. 平台分叉实现 SHOULD NOT 依赖响应式断点作为主机制。

### 4.6 跨端布局模型（统一文档，分端约束）

1. 移动端 MUST 使用三分区主导航语义：`卡片` / `池` / `设置`，且 `卡片` 为默认落点。
2. 桌面端 MUST 使用三栏工作区语义：主导航区 / 列表区 / 详情与编辑区。
3. 两端布局可不同，但页面语义 MUST 一一对应：`卡片`、`池`、`设置` 三域不可缺失。
4. 从 `设置` 返回主任务路径（卡片或池）MUST 在一步可达范围内。
5. 平台选择策略 MUST 在目标平台构建与运行时确定对应 UI，不要求同一端内随窗口宽度切换主结构。

### 4.7 输入交互映射

1. 移动端 SHOULD 以触控路径为主：点击进入、长按动作菜单、手势返回触发离开保护判定。
2. 桌面端 SHOULD 支持键鼠效率路径：右键菜单、双击打开、方向键移动、`Enter` 打开、`Cmd/Ctrl+N` 新建、`Cmd/Ctrl+S` 保存。
3. 跨端输入方式可不同，但同一动作的完成结果 MUST 一致。

### 4.8 关键流程语义（主页/卡片/数据池/设置/池域同步反馈）

1. 主页默认进入 `卡片` 页，且创建/加入池入口 MUST 仅在 `池` Tab 的未加入态提供。
2. 主页返回 MUST 满足双段返回语义；退出前 MUST 显示确认。
3. 卡片管理 MUST 覆盖新建、检索、编辑、保存、删除/恢复，并提供保存反馈与离开保护。
4. 数据池管理 MUST 覆盖创建、加入、审批、退出、解散（owner）；失败 MUST 可恢复。
5. 设置页 MUST 保持主路径可达，不得形成“只能返回不能直达”的阻断路径。
6. 池域内同步异常反馈 MUST 可见、可处理、且不阻断本地卡片读写。

### 4.9 平台路由映射（实现对照）

1. 两端可使用不同路由命名与组织方式，但 MUST 映射到同一业务语义页面。
2. 任一端新增关键页面时，MUST 同步补齐另一端的语义映射与验收条款。

| 业务语义页 | 移动端 UI 路由语义 | 桌面端 UI 路由语义 |
| --- | --- | --- |
| 首屏入口 | 启动直达 `卡片` Tab（可一步切换到 `池` Tab） | 启动直达工作台 `卡片` 区（可一步切换到 `池` 区） |
| 卡片主工作区 | `卡片` Tab（列表 -> 编辑） | 工作台 `卡片` 区（列表/详情并置） |
| 池管理 | `池` Tab（未加入/已加入/异常） | 工作台 `池` 区（状态与审批并置） |
| 设置 | `设置` Tab | 工作台 `设置` 区 |
| 退出确认 | 卡片根页系统返回弹窗 | 卡片根区返回意图弹窗 |
| 同步异常处理 | `池` 页内局部反馈与恢复动作 | 工作台 `池` 区内局部反馈与恢复动作 |

## 5. 视觉中约束

### 5.1 色彩系统

1. 系统 MUST 定义语义色位：主色、成功、警告、错误、信息、背景、边框、文本主/次级。
2. 系统 SHOULD 保证文本与背景达到可读对比。

#### 5.1.1 语义色 Token 命名（强约束）

系统 MUST 至少提供以下语义色 Token（命名可等价，但语义不可缺失）：

1. `color.primary`
2. `color.success`
3. `color.warning`
4. `color.error`
5. `color.info`
6. `color.bg.canvas`
7. `color.bg.surface`
8. `color.border.default`
9. `color.text.primary`
10. `color.text.secondary`
11. `color.action.danger`
12. `color.state.degraded`

#### 5.1.2 状态到色彩的映射（强约束）

1. 成功反馈 MUST 使用 `success` 语义色。
2. 可恢复风险提示 MUST 使用 `warning` 语义色。
3. 失败反馈与危险动作 MUST 使用 `error` 或 `action.danger` 语义色。
4. 信息提示 MUST 使用 `info` 语义色。
5. 同步降级（`degraded`）MUST 使用 `state.degraded`，且 FORBIDDEN 与致命错误色完全同态，避免误导。

#### 5.1.3 明暗主题最小色盘（可落地建议）

为降低实现漂移，系统 SHOULD 提供最小 Light/Dark 语义色盘。以下为参考值（可等价替换）：

| Token | Light 建议 | Dark 建议 |
| --- | --- | --- |
| `color.primary` | `#3B82F6` | `#60A5FA` |
| `color.success` | `#16A34A` | `#22C55E` |
| `color.warning` | `#D97706` | `#F59E0B` |
| `color.error` | `#DC2626` | `#F87171` |
| `color.info` | `#0284C7` | `#38BDF8` |
| `color.state.degraded` | `#EA580C` | `#FB923C` |
| `color.bg.canvas` | `#F8FAFC` | `#0F172A` |
| `color.bg.surface` | `#FFFFFF` | `#111827` |
| `color.border.default` | `#E2E8F0` | `#374151` |
| `color.text.primary` | `#0F172A` | `#F8FAFC` |
| `color.text.secondary` | `#475569` | `#94A3B8` |
| `color.action.danger` | `#B91C1C` | `#EF4444` |

#### 5.1.4 对比度与可读性门槛

1. 正文文本与背景对比度 SHOULD `>= 4.5:1`。
2. 大号文本（等价于 18pt 以上或加粗大号）与背景对比度 SHOULD `>= 3:1`。
3. 仅依赖颜色传达关键状态为 FORBIDDEN；关键状态 MUST 同时包含文本或图标语义。

#### 5.1.5 关键组件色彩规则（实现对照）

1. 主按钮默认态 SHOULD 使用 `color.primary`，禁用态 SHOULD 降低对比并保持可识别。
2. 危险动作按钮 MUST 使用 `color.action.danger`，并与普通主按钮在颜色语义上可区分。
3. Banner/提示条 MUST 按状态语义选色（`success/warning/error/info/degraded`），不得复用单一颜色覆盖全部状态。
4. 输入框错误态边框/文本提示 MUST 使用 `error` 语义色，且错误提示文案需同时可见。

### 5.2 字体与层级

1. 系统 MUST 定义文本层级（标题/正文/辅助/按钮）。
2. 系统 SHOULD 定义最小字号与行高边界。

### 5.3 间距与密度

1. 系统 MUST 使用统一间距节奏。
2. 系统 SHOULD 提供跨端密度策略，避免信息拥挤或过稀。

### 5.4 动效

1. 动效 MUST 服务于状态变化与操作反馈。
2. 动效时长与节奏 SHOULD 处于可感知但不拖慢操作的范围。

### 5.5 响应式

1. 本规格不要求基于断点的响应式主布局切换。
2. 移动端与桌面端各自 UI 中，关键按钮与关键信息 MUST 可达。
3. 两套 UI 的视觉可分化演进，但改变关键流程语义为 FORBIDDEN。

### 5.6 布局度量基线（可落地）

以下度量用于约束布局一致性与可实现性，属于“可观察布局行为”约束；允许在不破坏语义前提下做平台微调。

#### 5.6.1 间距与圆角 Token

1. 间距 Token MUST 至少包含：`4 / 8 / 12 / 16 / 20 / 24 / 32`（单位等价于逻辑像素）。
2. 页面主容器左右内边距：
   - 移动端 SHOULD 使用 `16`。
   - 桌面端 SHOULD 使用 `24~32`。
3. 卡片类容器圆角 SHOULD 在 `10~16` 区间。
4. 同一页面同级容器间距 MUST 使用同一 Token，不允许混用任意值造成节奏漂移。

#### 5.6.2 触控/点击目标

1. 移动端可点击主目标高度 MUST `>= 44`。
2. 桌面端可点击主目标高度 SHOULD `>= 32`，高频主动作 SHOULD `>= 36`。
3. 危险动作按钮与普通动作按钮 MUST 在视觉上可区分（语义色或描边/文案层级）。

#### 5.6.3 字体层级建议值

1. 页面主标题 SHOULD 在 `20~24`。
2. 区块标题 SHOULD 在 `16~18`。
3. 正文 SHOULD 在 `14~16`。
4. 辅助信息 SHOULD 在 `12~14`，但 MUST 保持可读对比。

### 5.7 双端主布局蓝图

#### 5.7.1 移动端主布局（3 Tab）

1. 顶层结构 MUST 为：`顶部栏（可选） + 内容区 + 底部 Tab`。
2. 底部 Tab MUST 固定为 `卡片 / 池 / 设置`，且 `卡片` 为默认落点。
3. 页面内容区滚动时，关键主动作（如新建卡片）MUST 始终可达（悬浮按钮或顶部动作位均可）。
4. 同步异常反馈 SHOULD 仅在 `池` 页内容区内呈现，且不得遮挡主操作超过一个可视区。

#### 5.7.2 桌面端主布局（三栏）

1. 顶层结构 MUST 为：`左导航栏 + 中列表栏 + 右详情/编辑栏`。
2. 三栏宽度 SHOULD 使用区间约束而非固定像素：
   - 左栏：`220~280`
   - 中栏：`320~420`
   - 右栏：剩余空间，且 SHOULD `>= 480`
3. 窗口缩小时 MAY 压缩栏宽，但 FORBIDDEN 隐藏主语义区（卡片、池、设置）导致不可达。
4. 桌面端高频动作 SHOULD 具备键盘路径（见 4.7），且操作结果与移动端一致。

### 5.8 页面级布局规格（移动端/桌面端）

#### 5.8.1 主页默认入口

1. 首屏 MUST 默认进入 `卡片`，并可一步切换到 `池` 与 `设置`。
2. 创建或加入池入口 MUST 仅在 `池` 页未加入态呈现，不得在应用首屏单独分流。
3. 桌面端焦点顺序 MUST 为自上而下、从左到右，`Enter` 触发当前焦点默认动作。

#### 5.8.2 卡片列表页

1. 页面 MUST 包含：搜索入口、卡片列表、新建入口。
2. 空态时 MUST 显示“为何为空 + 下一步动作（创建第一张卡片）”。
3. 移动端列表项高度 SHOULD `>= 64`；桌面端列表行高 SHOULD `>= 44`。
4. 桌面端列表区与详情区 SHOULD 支持并置；移动端可采用列表->详情分层导航。

#### 5.8.3 卡片编辑页

1. 编辑页 MUST 包含标题输入区、正文编辑区、保存状态反馈区。
2. 保存状态反馈 MUST 在不离开页面的前提下可见（如顶部状态字样或页内提示区）。
3. 当存在未保存改动触发返回时，离开保护弹层 MUST 覆盖当前编辑上下文并提供三分支动作。

#### 5.8.4 数据池列表页

1. 池页 MUST 在布局上可区分三态：未加入、已加入、异常。
2. 未加入态 MUST 在首屏提供 `创建池` 与 `扫码加入`。
3. 已加入态 MUST 可见：成员信息、同步状态、退出池入口。
4. 管理员可见审批区 SHOULD 与成员区分组展示，避免动作混淆。
5. 错误提示区 MUST 与动作按钮相邻，确保“看见错误即可执行下一步”。

#### 5.8.5 设置页

1. 设置页 MUST 为当前版本的空白占位页。
2. 从设置页到卡片页/池页 MUST 一步可达，不得依赖多级返回链。
3. 设置页 MUST NOT 暴露池相关入口、问题排查入口或额外分流入口。

#### 5.8.6 池域内同步反馈

1. 异常提示 MUST 提供语义文案 + 立即动作（`retry`/`reconnect`），且仅在 `池` 域内展示。
2. 异常提示层级 SHOULD 高于池域普通信息提示，但不应遮挡核心编辑区域输入焦点。
3. `degraded` 态提示 MAY 常驻于池域，但 MUST 不抢占阻断式模态窗口。

### 5.9 视觉一致性检查点（设计评审用）

1. 同一业务语义在双端 MUST 满足“入口可找到、反馈可看见、失败可恢复”。
2. 同一页面中的主次按钮层级 MUST 稳定，不得在不同状态反转语义优先级。
3. 危险动作（删除/退出/解散）在双端 MUST 使用统一语义文案与确认结构。
4. 列表、表单、弹层三类容器 SHOULD 复用统一间距与圆角 Token。

### 5.10 无障碍（A11y）约束

#### 5.10.1 键盘与焦点

1. 桌面端所有关键路径操作 MUST 可通过键盘完成（至少覆盖导航、确认、取消、保存、返回）。
2. 焦点顺序 MUST 与视觉阅读顺序一致，且不得跳过关键可交互控件。
3. 焦点可见性 MUST 明确可辨（描边、底色或等价视觉提示）。
4. 弹层出现时，焦点 MUST 进入弹层；弹层关闭后 SHOULD 返回触发控件或等价位置。

#### 5.10.2 语义与状态传达

1. 关键状态变化（成功、失败、提交中）MUST 以文本可读方式表达，不得仅依赖颜色。
2. 图标按钮 MUST 具备可理解的文本标签或可等价读出的语义标签。
3. 错误提示 MUST 包含“问题描述 + 下一步动作”，并与对应输入/动作区域建立可感知关联。

#### 5.10.3 目标尺寸与手势替代

1. 移动端关键点击目标 MUST 满足 5.6.2 的最小尺寸要求。
2. 依赖手势的关键操作 SHOULD 提供显式替代入口（例如按钮、菜单项）。
3. 仅通过复杂手势（长按拖拽组合）完成关键流程为 FORBIDDEN。

### 5.11 国际化与文案伸缩约束

1. 关键按钮与导航项 MUST 支持文本扩展（建议按 1.3~1.5 倍长度预留）。
2. 文案超长时 SHOULD 优先换行，其次截断；截断时 MUST 保留可理解性。
3. 表单标签、错误提示、空态说明 MUST 避免硬编码固定宽度导致语义丢失。
4. 布局在长文案场景下 MAY 调整间距与行数，但 FORBIDDEN 造成主动作不可达。
5. 若后续引入 RTL 语言，导航顺序与返回语义 MUST 保持一致业务含义。

## 6. 状态与反馈模型

系统界面状态 SHOULD 至少覆盖以下语义：

1. 初始化。
2. 加载中。
3. 可交互。
4. 提交中。
5. 成功反馈。
6. 失败反馈。
7. 空态。
8. 禁用态。

每个状态 SHOULD 定义触发条件、用户可执行动作、状态退出条件。

### 6.1 启动与主页状态流

1. 启动流程 SHOULD 满足：`Booting -> LocalReady -> PoolProbing`。
2. 达到 `LocalReady` 后，本地卡片读写 MUST 可用。
3. 池状态探测失败时，MUST 进入可恢复异常态，不得回退为不可用主界面。

### 6.2 池流程状态流

1. 加入池 SHOULD 满足：`JoinIdle -> JoinRequesting -> JoinSuccess | JoinFailed(code)`。
2. 审批 SHOULD 满足：`Pending -> Approving/Rejecting -> Done | Failed`。
3. 退出池 SHOULD 满足：`ExitConfirm -> Exiting -> ExitDone | ExitPartialFailed`。
4. `ExitPartialFailed` MUST 提供明确恢复动作（如重试清理）。

### 6.3 同步异常反馈分级

1. 系统 SHOULD 区分弱提示与高亮提示。
2. 异常态 MUST 提供 `retry` 或语义等价动作。
3. 需要重连的异常 MUST 提供 `reconnect` 或语义等价动作。
4. `degraded` 态 MAY 降级提示，但 FORBIDDEN 阻断本地编辑。

### 6.4 场景规格模板（用于实现与验收对照）

以下模板用于将本规格中的关键流程条款映射为可执行的场景规格。模板用于约束“可观察行为”，不约束具体技术实现。

#### 6.4.1 模板字段

1. `入口`：用户可触发入口（按钮、Tab、菜单、快捷键、系统返回等）。
2. `前置条件`：场景成立所需的可观察系统状态。
3. `主流程`：按步骤编号描述期望交互流程。
4. `异常分支`：至少 1 条失败或中断路径。
5. `恢复动作`：用户在失败后可执行的下一步动作。
6. `退出条件`：场景完成、取消或中止的可观察终态。
7. `可观察结果`：页面文本、对话框、路由变化、状态提示等黑盒可见结果。

#### 6.4.2 模板约束

1. 每个关键流程场景 MUST 至少包含 1 条主流程与 1 条异常分支。
2. 每个异常分支 MUST 绑定至少 1 个恢复动作。
3. 每个步骤 MUST 可被黑盒测试验证，不得依赖内部实现细节。
4. 当跨端触发方式不同，场景模板 MAY 分端描述触发入口，但 MUST 共享同一结果语义。

### 6.6 错误语义与恢复动作映射（加入池）

| --- | --- | --- | --- |
| `POOL_NOT_FOUND` | 目标池不存在或来源失效 | 重新获取池信息 | `retry` |
| `INVALID_POOL_HASH` | 池标识无效 | 重新扫码 | `reconnect` |

### 6.7 性能预算与反馈时延（宽松基线）

本节提供“可用性优先”的宽松预算，用于避免明显卡顿或无反馈状态；不追求高压性能指标。

1. 用户触发关键动作后，界面 SHOULD 在 `<= 800ms` 内出现“处理中/反馈中”可见提示。
2. 页面级主内容首次可交互时间 SHOULD 在 `<= 3s`（常规设备、常规本地数据规模）范围内。
3. 本地保存动作 SHOULD 在 `<= 2s` 内给出成功/失败反馈；超过该阈值 MUST 持续显示处理中状态。
4. 网络相关流程（加入池、审批、同步重连）允许更长耗时，但 MUST 保持可取消或可返回路径。
5. 连续动画时长 SHOULD 控制在 `120~300ms` 常用区间；超过 `500ms` 的动效仅可用于明确过渡语义。
6. 若预算未达标，MUST 优先保证“可见反馈 + 可恢复路径”，而非静默等待。

## 7. 验收标准（黑盒）

验收场景 SHOULD 使用 Given/When/Then 表达。

### 7.1 最小验收集

1. Given 关键流程页面，When 用户触发返回，Then 返回语义与预期一致且不丢失未确认数据。
2. Given 危险动作，When 用户触发，Then 系统先展示确认并明确影响范围。
3. Given 提交动作执行中，When 观察界面，Then 可见处理中反馈且不可误触重复提交。
4. Given 提交失败，When 观察界面，Then 可见区分错误语义与恢复动作。
5. Given 空列表场景，When 首次进入页面，Then 展示空态原因与下一步动作。
6. Given 同一业务动作在移动端与桌面端执行，When 对比结果，Then 结果语义与恢复路径一致。
7. Given 用户在设置页，When 切换主导航，Then 一步可达卡片页与池页。
8. Given 用户在主页非卡片页，When 触发系统返回，Then 先回到卡片页而非直接退出。
9. Given 用户在卡片根页，When 触发系统返回，Then 显示退出确认且可取消。
10. Given 加入池失败（任一稳定错误语义），When 观察界面，Then 可见“发生了什么 + 下一步动作”。
11. Given 同步处于 degraded，When 用户编辑并保存卡片，Then 本地保存成功且同步异常仅以提示呈现。

### 7.2 判定原则

1. 验收 MUST 以外部可观察行为为准。
2. 验收依赖组件名、函数名、日志文本等实现细节为 FORBIDDEN。

### 7.3 场景化最小验收集（主页/卡片/数据池/设置/池域同步反馈）

#### 主页默认进入卡片与返回语义

1. Given 用户首次启动应用，When 启动完成，Then 首屏默认进入卡片主工作区。
2. Given 用户位于卡片页，When 切换到池页，Then 可在未加入态看到创建/加入池入口。
3. Given 用户位于主页非卡片页，When 触发系统返回，Then 先回到卡片根页而非退出应用。
4. Given 用户位于卡片根页，When 触发系统返回，Then 出现退出确认弹窗且可取消。

#### 卡片管理（CRUD + 保护）

1. Given 用户位于卡片页，When 新建并保存卡片，Then 可见“本地已保存”反馈且列表可见新卡片。
2. Given 存在已保存卡片，When 执行搜索，Then 仅在 Active Note 范围内按标题+正文子串不区分大小写匹配。
3. Given 编辑页存在未保存改动，When 尝试离开页面，Then 出现离开保护并提供“保存并离开/放弃更改/取消”三分支。
4. Given 同步异常存在，When 用户继续编辑并保存，Then 本地保存成功且异常仅以提示形式呈现。

#### 数据池管理（创建/加入/审批/退出）

1. Given 用户未加入池，When 发起创建池或加入池，Then 进入对应流程状态并有可观察反馈。
2. Given 加入池失败（任一稳定错误码），When 观察界面，Then 显示“发生了什么 + 下一步动作”。
3. Given 存在待审批请求且当前用户有审批权限，When 执行通过/拒绝，Then 请求项状态立即变化且失败可重试。
4. Given 用户执行退出池，When 发生部分清理失败，Then 明确显示“部分清理失败”并提供重试清理动作。

#### 设置页可达性

1. Given 用户位于设置页，When 切换主导航到卡片页，Then 一步可达卡片主工作区。
2. Given 用户位于设置页，When 切换主导航到池页，Then 一步可达池管理页（未加入态或已加入态）。

#### 池域内同步反馈

1. Given 出现同步异常，When 观察池页界面，Then 可见局部异常提示并可触发 `retry` 或 `reconnect`。
2. Given 处于 `degraded`，When 用户执行本地卡片编辑保存，Then 本地流程保持可用且不被阻断。

### 7.4 场景到测试的映射要求

1. 每个关键流程场景 MUST 至少映射 1 条成功路径测试。
2. 每个关键流程场景 MUST 至少映射 1 条失败或中断路径测试。
3. 池相关场景 MUST 覆盖至少 1 条稳定错误码失败路径。
4. 映射测试 MUST 断言行为结果（路由、状态文本、弹窗、列表变化、错误反馈），仅断言组件存在为 FORBIDDEN。

### 7.5 A11y / 性能 / i18n 最小验收补充

1. Given 桌面端关键页面，When 用户仅使用键盘执行主流程，Then 可完成导航、确认、取消、保存与返回。
2. Given 关键动作触发，When 动作进行中，Then 在预算时间内可见处理中反馈，且不可出现“无反馈长等待”。
3. Given 文案长度扩展（至少 1.3 倍），When 渲染关键页面，Then 主动作仍可见可达且语义不丢失。
4. Given 错误状态出现，When 用户观察界面，Then 同时可见错误语义文本与恢复动作，而非仅颜色变化。

## 8. 兼容与演进

1. 交互语义变更影响可观察行为时，MUST 先更新本规格后更新实现。
2. 视觉细节可在中约束边界内演进，但 MUST 不破坏交互强约束。
3. 新增关键流程 MUST 同步补充本规格中的交互与验收条款。
