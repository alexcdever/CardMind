## 1. 目的与范围

本规格定义 CardMind 项目级架构的行为与结构约束，作为项目级上位架构约束来源。

本规格覆盖：

1. Flutter、Rust、FRB、`LoroDoc`、`SQLite` 的职责边界。
2. 唯一真实信源与查询侧读模型的定义。
3. 读写分离链路与查询路径约束。
4. 一致性、投影、同步失败的语义边界。
5. 可用于黑盒验收的项目级架构场景。

本规格不覆盖：

1. 具体 API 名称、参数签名与目录结构。
2. 迁移顺序、任务拆解与实施步骤。
3. 分域私有业务流程细节。

## 2. 术语

1. Frontend：承担界面渲染、交互编排、展示状态与后端调用的前端层。
2. Backend：承担领域规则、业务写入、同步与稳定契约输出的后端层。
3. Language Boundary：跨语言调用的边界层，仅承担数据传输与调用桥接职责。
4. Source of Truth：用于判定业务事实的唯一真实信源。
5. Write Model：承载业务事实写入的模型层。
6. Read Model：为查询、筛选、排序与展示优化的读取模型层。
7. Projection：从写模型变化推导并更新到读模型的过程。
8. Sync Convergence：多参与方状态经过同步后收敛为一致可观察结果的过程。

## 3. 规范性规则

本节采用以下术语强度：

1. MUST：必须满足。
2. FORBIDDEN：禁止。
3. SHOULD：建议满足，若不满足需有明确理由。
4. MAY：可选能力。

## 4. 顶层架构职责

### 4.1 Flutter 前端职责

1. Flutter MUST 作为前端层，只承担 UI 渲染、交互流程编排、展示状态管理与后端调用。
2. Flutter MAY 持有页面态、表单态与展示缓存，但这些状态不得被视为领域真实信源。
3. Flutter FORBIDDEN 直接承担领域写规则、持久化真源控制或同步收敛判定职责。

### 4.2 Rust 后端职责

1. Rust MUST 作为后端层，承担领域规则、业务写入、投影驱动、同步与稳定数据契约输出。
2. Rust MUST 掌管卡片笔记与数据池元数据相关的业务不变量。
3. Rust SHOULD 对 Flutter 暴露稳定的用例边界、查询边界或等价后端契约，而不是泄露内部存储细节。

### 4.3 FRB 语言边界职责

1. FRB MUST 作为 Language Boundary，只承担跨语言请求、响应与错误传递。
2. FRB FORBIDDEN 演变为独立业务层。
3. 除边界传输所需的最小转换外，FRB FORBIDDEN 复制领域规则或长期承载业务分支。

## 5. 读写分离与数据流

### 5.1 真源与读模型

1. `LoroDoc` MUST 是卡片笔记与数据池元数据的唯一真实信源。
2. `SQLite` MUST 仅作为查询侧读模型，不得作为业务真源。
3. 任意业务事实变更 MUST 先落入 `LoroDoc`。

### 5.2 写入链路

1. 所有业务写入 MUST 通过 Rust 后端进入 `LoroDoc`。
2. Flutter 直接写入 `LoroDoc` 为 FORBIDDEN。
3. 直接写入 `SQLite` 来表达业务事实为 FORBIDDEN。

### 5.3 查询链路

1. 面向产品行为的查询 MUST 走 `SQLite`。
2. Flutter 或 Rust 为展示、筛选、搜索、排序直接把 `LoroDoc` 当查询主路径为 FORBIDDEN。
3. 运维、诊断、修复或等价内部技术流程 MAY 读取 `LoroDoc` 或其他内部状态，但不得绕过本规格定义的产品查询链路。
4. 查询侧辅助缓存 MAY 存在，但不得改变“产品查询结果以 `SQLite` 为准”的约束。

### 5.4 投影与同步链路

1. `LoroDoc` 变更 MUST 通过订阅机制、投影流程或等价链路同步到 `SQLite`。
2. 外部同步结果 MUST 先进入 `LoroDoc`，再驱动 `SQLite` 收敛。
3. 禁止绕过 `LoroDoc -> SQLite` 链路直接改写 `SQLite` 表达同步后的业务事实。

## 6. 中层结构约束

1. Flutter 访问 Rust MUST 通过明确的前端调用边界。
2. Flutter FORBIDDEN 在 Dart 侧复制 Rust 已掌管的业务写主路径。
3. Flutter FORBIDDEN 同时维护与 Rust 冲突的第二套领域规则判定。
4. Rust 返回给 Flutter 的结果 SHOULD 以稳定 DTO、稳定错误码或等价稳定契约表达。
5. Flutter 页面与组件 MAY 自行组合展示态，但业务事实判断 MUST 以后端写入结果与 `SQLite` 查询结果为准。

## 7. 一致性、错误与恢复原则

1. 业务写入成功 MUST 以 `LoroDoc` 写入成功为判定依据。
2. `SQLite` 投影失败不得伪装成业务写失败。
3. 同步失败不得伪装成业务写失败。
4. 系统 MAY 存在短暂查询延迟或短暂同步延迟，但最终可观察查询结果 MUST 与 `LoroDoc` 一致。
5. 系统 MUST 能区分至少以下状态语义：业务写成功、投影未收敛、同步未收敛。
6. 可恢复失败 MUST 提供至少一个明确恢复动作，例如重试投影、重试同步或重新查询当前状态。

## 8. 禁止事项

1. Flutter 作为长期业务真源为 FORBIDDEN。
2. `SQLite` 作为业务写入真源为 FORBIDDEN。
3. 在 Flutter、FRB、Rust 三层重复复制同一领域写规则为 FORBIDDEN。
4. 绕过 Rust 直接形成客户端业务写主路径为 FORBIDDEN。
5. 为长期兼容保留双业务写架构主路径为 FORBIDDEN。

## 9. 验收标准（黑盒）

验收用例 SHOULD 使用 Given/When/Then 表达。

### 9.1 最小验收集

1. Given 用户创建卡片笔记或修改数据池元数据，When 业务动作成功完成并且查询侧收敛完成，Then 用户可经由 `SQLite` 查询观察到该结果。
2. Given 新的业务事实已经被系统接受，When 投影收敛完成，Then 产品查询结果可仅通过 `SQLite` 观察到最新状态。
3. Given 一次业务写入成功但同步失败，When 用户查看结果反馈，Then 用户看到“已保存但未同步完成”的语义，而不是业务失败语义。
4. Given 一次业务写入成功但投影暂未完成，When 用户在短暂延迟后重新查询，Then 查询结果最终收敛为一致状态。
5. Given Flutter 页面触发业务写操作，When 用户随后查询该结果，Then 该结果遵守本规格定义的 Rust 写入与 `SQLite` 查询链路语义。

### 9.2 判定原则

1. 验收 MUST 以外部可观察行为为准。
2. 验收依赖类名、函数名、表结构、日志文本等实现细节为 FORBIDDEN。

## 10. 兼容与演进

1. 分域规格可细化领域对象与行为，但若涉及项目级架构边界，MUST 遵守本规格。
2. 若实现行为变更影响本规格定义的外部架构约束，MUST 先更新本规格后更新实现。
3. 临时兼容路径若确有必要，SHOULD 明确记录其替代目标与删除边界，并不得改变本规格定义的长期目标架构。
