# Coding Style Standard

## 基本要求

- 先参考现有实现，再决定是否引入新写法
- 保持命名、目录结构、接口形状与仓库现有模式一致
- 优先显式写法，避免隐式行为、魔法数和难追踪的状态传递

## 命名与结构

- Dart 使用驼峰命名，Rust 使用 snake_case
- 只有在复用收益明确时才抽象公共层；默认优先做最小正确改动
- 单个文件和函数应保持单一职责，避免把无关职责堆进同一入口

## 注释与可读性

- 注释只解释不直观的约束、原因或边界，不重复代码字面含义
- 若 lint、格式化或语言惯例已能表达规则，不再重复造一层文档约束
- 行为复杂处优先通过更清晰的命名和结构提升可读性，而不是依赖长注释

## 与工具的关系

- Dart/Flutter 风格基线以 `analysis_options.yaml` 和 `flutter_lints` 为准
- 本文档只补充 lint 不会表达、但对团队协作和 AI 改动质量有影响的项目级偏好
