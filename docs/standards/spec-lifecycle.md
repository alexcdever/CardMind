# Spec 生命周期规范

## 目标

- 定义什么是 `spec`
- 定义何时必须更新 `docs/specs/`
- 明确 `spec`、`plan`、`standard` 的边界
- 为 AI 和人工协作提供统一执行入口

## 文档定义

- `spec`：系统当前已确认的正式行为、约束、验收标准与非目标
- `plan`：某次变更的设计思路、任务拆解、风险与验证策略
- `standard`：长期稳定、跨任务复用的工程与协作规则

判断标准：

- `spec` 回答“系统应该如何表现”
- `plan` 回答“本次改动如何落地”
- `standard` 回答“长期应遵守什么规则”

## Spec 更新触发规则

- 只有当正式行为已确认发生变化时，才更新 `docs/specs/`
- 仅因为准备开始实现，不构成更新 `docs/specs/` 的理由
- 未确认的需求、探索中的方案、临时设计备选项，不得直接写入 `docs/specs/`

通常需要更新 `docs/specs/` 的情形：

- 新功能
- 用户可感知的行为变化
- 数据模型或状态转换规则变化
- 错误处理预期变化
- Flutter、Rust、FRB 或存储/同步契约变化

通常不需要更新 `docs/specs/` 的情形：

- 修复既有预期行为的 bug
- 纯重构
- 测试补齐
- lint、构建、脚本修复
- 仅增加自动化锚点、注释或目录索引

如果实施过程中发现任务实际上改变了正式行为，应立即回退到规格判断，补充更新 `docs/specs/` 后再继续。

## 任务分类模型

### A 类：正式行为变更

示例：

- 新功能
- 用户可见行为变化
- 数据规则变化
- 同步语义变化
- Flutter/Rust/FRB 合同变化

处理要求：

- 先澄清并确认行为变化
- 更新 `docs/specs/`
- 后续实现、验证与交付流程按 `docs/standards/ai-collaboration.md` 执行

### B 类：范围明确的局部行为调整

示例：

- 局部 UX 增强
- 中等规模规则补强
- 范围清晰的行为加固

处理要求：

- 先判断是否真正改变正式行为
- 仅在正式行为变化时更新 `docs/specs/`
- 后续实现、验证与交付流程按 `docs/standards/ai-collaboration.md` 执行

### C 类：恢复性或纯工程任务

示例：

- 恢复既有预期行为的 bugfix
- 重构
- 测试补齐
- lint 或构建修复
- 文档索引与自动化锚点补齐

处理要求：

- 确认任务不重定义产品行为
- 可直接进入实现与验证，具体流程按 `docs/standards/ai-collaboration.md` 执行
- 若实现暴露真实行为变化，升级为 A/B 类重新处理

## 边界要求

- `docs/specs/` 不写任务顺序、技术拆解、临时方案比较
- `docs/plans/` 不作为当前长期行为真相源
- `docs/standards/` 不承载一次性任务编排
- `AGENTS.md` 只做入口与引用，不复制标准正文

## 目标不清时的处理

- 若行为意图不清、文档互相冲突或验收标准不完整，先停止实现并澄清
- 不允许通过“先写进 spec 再说”来替代确认
- 当计划与规格冲突时，以已确认的 `docs/specs/` 为准；若 `spec` 本身不清晰，需先补确认
