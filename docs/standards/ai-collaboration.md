# AI 协作规范

本文档定义 AI 驱动开发模式中的协作标准，适用于所有功能开发场景。

## 概述

本项目采用**双代理对抗模式**进行开发，通过实现者与审查者的持续对抗，确保代码质量、测试覆盖和设计符合规范。

## 双代理角色定义

### 实现者（Implementer）

**职责**：按 TDD 规范编写代码，使功能通过测试并达到可交付标准。

**工作方式**：
1. 阅读设计文档、计划文档、规格文档，明确目标和约束
2. 按 TDD 三阶段（红-绿-蓝）循环开发
3. 每次提交代码供审查者检验
4. 根据审查者反馈进行重构，直到通过审查

**约束**：
- 必须遵循现有代码风格和架构模式
- 所有代码修改必须通过测试验证
- 重构时不得破坏已有功能

### 审查者（Reviewer）

**职责**：审查实现者的代码，确保其符合设计规范和质量标准。

**评判依据**（按优先级排序）：
1. **规格文档**：代码是否实现了规格定义的行为
2. **设计文档**：架构和设计决策是否被正确落实
3. **计划文档**：任务执行是否符合预定方案
4. **质量规范**：
   - 测试有效性：测试是否真正验证了业务逻辑
   - 边界覆盖：关键边界条件是否被测试覆盖
   - 代码可读性：命名、结构、注释是否清晰
   - 开闭原则：扩展是否开放，修改是否封闭

**输出**：
- 通过：确认代码达标，可进入下一阶段或提交
- 不通过：列出具体问题，提供改进建议，返回重构

## 对抗循环机制

### 循环流程

```
实现者提交代码 → 审查者检验 → 不通过 → 实现者重构 → 审查者再检验
                                    ↓ 通过
                                进入下一阶段/提交
```

### 停止条件

**正常停止**：
- 审查者确认代码达标
- 当前 TDD 阶段（红/绿/蓝）完成且通过审查
- 所有任务完成，质量门禁通过

**异常停止（需用户介入）**：
- 循环超过 **5 次**仍未通过审查（可能存在设计分歧）
- 审查者与实现者对规格文档理解不一致
- 遇到无法解决的阻塞（依赖缺失、环境异常等）
- 发现规格文档本身存在矛盾或遗漏

## TDD 三阶段的对抗模式

每个阶段都必须经过完整的对抗循环：

### 红阶段（编写失败测试）

**实现者**：
- 编写注定失败的测试用例
- 测试应精确暴露业务逻辑缺口
- 运行测试确认失败（验证测试有效）

**审查者检验**：
- 测试是否真正测到了业务需求
- 测试失败原因是否合理
- 测试命名和结构是否清晰
- 是否覆盖了规格中的边界条件

### 绿阶段（编写实现代码）

**实现者**：
- 编写最简单的实现使测试通过
- 目标是让测试变绿，不必追求完美
- 运行测试确认通过

**审查者检验**：
- 实现是否确实使测试通过
- 实现是否满足规格定义的行为
- 是否有不必要的复杂度
- 是否引入了魔法数字或硬编码

### 蓝阶段（重构）

**实现者**：
- 以更合理、符合开闭原则且可读性更高的方式重构代码
- 保持所有测试通过的前提下优化设计
- 消除重复、改善命名、简化逻辑

**审查者检验**：
- 重构后所有测试是否仍通过
- 代码是否符合开闭原则
- 可读性是否有提升（命名、结构、注释）
- 重复是否被有效消除

## 质量门禁检查

### 自动检查项

实现者完成所有任务后，审查者触发质量门禁：

```bash
dart run tool/quality.dart <flutter|rust|all>
```

门禁检查包括：
1. 代码分析（lint）：风格、潜在问题
2. 测试执行：单元测试、集成测试
3. 边界扫描：识别未覆盖的边界条件

### 边界覆盖分析

读取 `/tmp/cardmind_test_boundary_report.md`，检查：

**高优先级边界**（必须覆盖）：
- 空值/空输入处理
- 异常和错误处理路径
- 集合的边界（空列表、越界）

**中优先级边界**（建议覆盖）：
- 异步状态转换（loading/error/success）
- 并发场景（如果适用）

**低优先级边界**（记录到待办）：
- 极端输入值（超长字符串、极大数字）
- 罕见的竞态条件

## 冲突解决

当实现者与审查者出现分歧时：

**第一优先级**：规格文档
- 以规格文档为准，若规格文档不明确，停止并请示用户

**第二优先级**：设计文档
- 以设计文档定义的架构和接口为准

**第三优先级**：计划文档
- 以计划文档的任务分解为准

**最后手段**：用户决策
- 若文档之间存在矛盾，或双方理解无法统一，停止开发并请示用户

## 环境隔离要求

### 工作分支

每个功能开发必须在独立的工作分支（worktree）中进行：
- 分支命名：`feature/<功能名>` 或 `fix/<问题描述>`
- 从干净的 main 分支创建
- 验证基线测试通过后再开始开发

### 代码提交

- 每个 TDD 阶段完成后提交一次
- 提交信息遵循 `docs/standards/git-and-pr.md`
- 保持提交原子性，便于回滚和审查

## 边界检查清单

| 层级 | 边界类型 | 检查项 | 优先级 |
|------|---------|--------|--------|
| Flutter | 空值/空输入 | 空字符串、空列表、null 处理 | 高 |
| Flutter | 异常处理 | try/catch、错误回调、错误提示 | 高 |
| Flutter | 异步状态 | loading/error/success 状态管理 | 中 |
| Flutter | 集合边界 | 空列表、越界访问、迭代安全 | 高 |
| Flutter | UI 响应式 | 布局断点（如 900px）、键盘弹出 | 中 |
| Flutter | 焦点管理 | 输入框焦点、快捷键冲突 | 中 |
| Rust | FFI 边界 | 参数验证、错误类型转换 | 高 |
| Rust | 并发安全 | Arc/Mutex 使用、数据竞争防护 | 高 |
| Rust | 错误处理 | Result/Option 正确处理、错误传播 | 高 |
| Rust | 资源管理 | Drop 实现、内存泄漏防护 | 高 |
| Rust | 异步边界 | async/await、Tokio 运行时正确使用 | 中 |
| 跨层 | 序列化边界 | JSON/Protobuf 解析错误处理 | 高 |
| 跨层 | 类型边界 | FFI 类型转换安全、大小端问题 | 高 |
