# AI 协作规范

## 目标

- 为人类与 AI 提供统一、可执行的协作协议
- 让任务复杂度与执行流程匹配，避免默认进入过重流程
- 保持目标、范围、验收、验证证据始终清晰

## 通用原则

- 开始实现前应先判断当前信息是否足以安全推进，并明确目标、范围和验收标准
- 是否需要更新 `docs/specs/`，按 `docs/standards/spec-lifecycle.md` 判断
- 任何任务都应优先复用现有代码、文档与约定
- 若遇到目标不清、文档冲突、验证持续失败或环境阻塞，应暂停并向用户反馈

## 分级执行模型

### 小任务

适用场景：

- 小范围修复
- 局部文档修订
- 机械性重命名
- 范围明确的工程小改动

执行顺序：

1. 判断当前信息是否足以安全推进；若会影响正式行为判断或方案分叉，先澄清目标与验收标准
2. 直接实现
3. 运行与改动范围匹配的验证
4. 交付结果与影响说明

### 中任务

适用场景：

- 有明确边界的功能调整
- 中等规模结构优化
- 需要多步落地但不涉及大范围系统重构

执行顺序：

1. 判断目标、范围、验收标准是否已足以支撑实现；若关键边界不清，先澄清
2. 必要时补充 `docs/plans/`
3. 实现
4. 自查或请求代码审查
5. 运行与影响范围匹配的验证

### 大任务

适用场景：

- 新功能或跨层行为变更
- 重要架构调整
- 涉及多个模块、多个阶段或高风险契约调整

执行顺序：

1. 澄清并确认目标、范围、验收标准
2. 先按 `docs/standards/spec-lifecycle.md` 判断是否发生正式行为变更；仅在发生时更新 `docs/specs/`
3. 编写并确认 `docs/plans/`
4. 实现
5. 审查
6. 执行完整质量门禁

## Agent 授权边界

### 可直接推进

- 恢复既有预期行为的 bugfix
- 范围明确的局部工程修正
- 测试补齐
- 文档索引维护
- 不改变正式行为的最小重构

### 必须升级确认

- 改变已确认的正式行为
- 引入新功能、新交互语义或新数据规则
- 扩大当前任务范围
- 存在多种合理方案，且会影响产品体验、验收口径或长期接口边界
- 文档、代码、测试之间存在无法自行消解的冲突

### 可自行收敛的局部判断

- 与现有模式一致的最小实现路径选择
- 局部命名与小范围结构整理
- 测试粒度、测试位置与验证命令选择
- 不改变行为前提下的代码组织调整
- 可通过现有代码、测试和文档直接验证的实现细节判断

## TDD 与审查策略

- 代码改动优先采用 TDD 循环，具体适用边界见 `docs/standards/tdd.md`
- 双代理对抗评审是高强度策略，仅在复杂、高风险或跨层任务中按需采用
- 小任务不要求默认进入双代理对抗模式
- 任何任务都必须有与其风险相称的验证与复核

## 环境与交付要求

- 是否使用独立 worktree，按任务风险、并发冲突风险和当前分支状态判断，不再作为所有任务的默认强制要求
- 不得在未说明验证结果的情况下宣称任务完成
- 交付时至少说明：改了什么、为什么这样改、运行了哪些验证、是否有剩余风险

## 冲突处理顺序

1. 用户明确要求
2. 已确认的 `docs/specs/`
3. 当前任务的 `docs/plans/`
4. `docs/standards/`
5. 仓库中的既有实现模式

如果以上内容之间存在冲突且无法自行消解，应停止并请求确认。
