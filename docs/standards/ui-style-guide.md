# UI 风格规范

## 目标与适用范围

- 定义 CardMind 的长期 UI 风格基线，作为页面设计、组件实现与原型复现的统一参考
- 本文档回答“界面长期应呈现什么视觉语言”，不回答具体页面“必须如何交互”或某次改动“如何落地”
- 适用于 Flutter 界面、设计原型、评审基线与后续页面规格编写

边界说明：

- 具体页面行为、状态流转、验收标准，属于 `docs/specs/`
- 某次改动的设计取舍与实施步骤，属于 `docs/plans/`
- 本文档只定义长期稳定、跨页面复用的视觉与组件风格规则

## 视觉方向

CardMind 的 UI 方向是“数字中庭（Digital Atrium）”。

核心特征：

- 以轻、透、稳为主，而不是传统去中心化产品常见的厚重工程感
- 通过版式留白、层级背景和有限强调色建立秩序，而不是依赖高密度边框与强阴影
- 在信息密集场景下仍保持阅读感，界面既像工具，也像经过整理的思考空间

设计原则：

- 优先营造“可呼吸的内容空间”，避免把界面做成控制台或监控面板
- 强调结构层次，而不是强调装饰
- 页面中的重点信息应通过字号、字重、背景层次和留白突出，不应依赖大量颜色竞争注意力

## 色彩与层级

### 主色方向

- 主色采用稳定的深青绿色系，作为品牌、主按钮、重要状态与关键路径强调
- 中性色采用偏冷的浅灰与浅青灰，用于背景、分层和承托正文
- 文本避免纯黑，优先使用柔和深色，减少压迫感

### 分层规则

- 禁止把 1px 实线边框作为默认分区手段
- 页面区块分层优先依赖背景色阶变化
- 推荐层级：
  - 基础画布：最浅背景
  - 区块层：略深一级的容器背景
  - 当前焦点卡片：白色或最高亮度容器

### 强调色使用

- 主色仅用于主操作、选中态、品牌点和高优先级状态
- 成功/连接中状态可使用偏青绿色或青蓝绿色系强调
- 非必要场景避免大面积高饱和色块

## 字体与文字层级

### 字体角色

- 标题与展示层可使用更具建筑感、几何感的字体，负责建立品牌气质和主层级
- 正文、说明、元数据优先使用高可读性的无衬线字体
- 同一页面内应明确区分“展示层”和“功能层”，不要让所有文字看起来同权重

### 文字层级

- 大标题用于页面 framing，不应用于高频重复组件
- 正文是默认阅读层，应保证足够留白和舒适行高
- 标签和状态字用于技术语义补充，例如同步状态、时间、密钥、节点信息

### 文本规则

- 避免纯黑文字
- 避免为了“科技感”在正文中大面积使用全大写
- 标签、小字说明可以使用更强的字重和更宽的字距，但必须控制范围

## 容器、边框、阴影与纹理

### 容器

- 卡片、面板、编辑器画布应通过圆角和层级背景形成柔和边界
- 内容容器默认保持较大内边距，避免内容贴边
- 需要突出焦点时，优先提升容器层级，而不是增加描边厚度

### 边框

- 默认不使用明显边框分隔主区块
- 仅在可访问性对比不足或组件边界不清时，使用低对比度“幽灵边框”
- 幽灵边框应弱感知，作用是补充边界，而不是主导视觉

### 阴影

- 普通组件不依赖传统重阴影建立层级
- 只有浮层、菜单、悬浮工具或模态等真正“漂浮”元素才使用环境阴影
- 环境阴影应柔和、扩散、低不透明度，避免 Material 默认那类厚重投影感

### 纹理

- 可以在大画布或编辑区域使用极轻的点阵、渐变或雾化质感，增强空气感
- 纹理必须退居背景，不能干扰文本可读性

## 核心组件风格约束

### 按钮

- 主按钮：使用主色填充，承担主路径动作
- 次按钮：使用浅色容器背景，不使用显性边框
- 文字按钮：不加底色，悬停或激活时通过浅背景反馈

规则：

- 主按钮优先出现在页面主路径上，一个局部区域通常只保留一个强主按钮
- 不要同时堆叠多个视觉权重相同的主按钮

### 输入与编辑器

- 输入区与编辑器优先表现为内容工作区，而不是传统表单控件
- 默认弱化输入框轮廓，重点通过排版、留白和当前行/当前状态提示建立可编辑感
- 当前同步或编辑态可以通过细条、高亮点或小状态标签表达

### 卡片与列表项

- 卡片之间优先通过留白分隔，不使用密集分割线
- 未选中态使用中性浅层背景
- 当前选中或焦点态使用更亮一层的容器背景
- 卡片内容内部保持明确的信息层次：标题、摘要、元数据、状态

### 状态标签与同步标签

- 状态标签应小而明确，承担“补充语义”而不是“主视觉”
- 活跃同步、在线、加密等状态可使用青绿系柔和标签
- 标签动画只用于必要状态，如同步中、连接中；动画应轻，不应分散注意力

### 浮动工具与悬浮操作

- 悬浮操作允许使用半透明浅底、背景模糊和轻阴影
- 这类元素应表现为“漂浮在内容上方”，但不能破坏内容主体阅读

## 页面布局倾向

- 倾向使用不完全对称的版式，让标题区、内容区、辅助区存在明确主次
- 标题、正文、辅助信息之间保留足够留白
- Desktop 与 Mobile 保持同一风格语言，但不要求机械复制布局
- Mobile 端优先保证垂直节奏、操作集中和底部导航清晰
- Desktop 端优先保证工作区、导航区和辅助信息区分层明确

## Do / Don't

### Do

- 用背景层次替代默认分割线
- 用留白和字重建立阅读秩序
- 在同步、成员、网络状态等技术信息上保持克制表达
- 在主流程入口上使用清晰但不过度张扬的强调

### Don't

- 不要把界面做成高对比、强描边、强阴影的工程控制台风格
- 不要让主色充满大面积背景
- 不要在同一区域并列多个同权重主操作
- 不要让状态标签、技术信息和正文争夺同一视觉优先级
- 不要把一次性原型装饰误当成长期规范

## 与规格文档的边界

- 本文档是长期 UI 风格规范，不直接定义页面行为真相
- 若某页面必须遵守特定布局、状态或交互规则，应写入对应 `docs/specs/`
- 若某次改动临时采用了偏离本规范的视觉方案，应在对应 `docs/plans/` 中说明原因与范围
- 当实现与本规范冲突时，优先级按仓库规则执行：
  - 用户明确要求
  - 已确认的 `docs/specs/`
  - 当前任务的 `docs/plans/`
  - 本文档及其他 `docs/standards/`

## 使用建议

- 新页面设计前，先用本规范校验视觉方向是否一致
- 做原型复现时，优先复现信息层次、容器层级和阅读节奏，再处理装饰细节
- 做代码实现时，优先沉淀为主题 token、间距规则和组件样式，而不是页面内重复手写
