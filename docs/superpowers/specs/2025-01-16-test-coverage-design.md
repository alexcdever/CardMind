## 1. 背景与目标

### 1.1 问题陈述

当前 CardMind 项目的自动化测试覆盖存在以下欠缺项：

1. **按数据池筛选卡片** - card-note.md 5.2 要求支持按所属数据池筛选，但测试中未见
2. **多成员协作一致性** - pool.md 6.2 要求池内成员修改后其他成员最终可见，需多成员场景验证
3. **并发幂等性** - pool.md 6.3 要求重复提交满足幂等预期，未测试
4. **完整 A11y 键盘路径** - ui-interaction.md 5.10 要求桌面端关键路径可通过键盘完成，覆盖不完整

**注**: 性能预算验证（ui-interaction.md 6.7）优先级较低，暂不纳入本次补全范围。

### 1.2 目标

通过分层递进的测试补全策略，确保所有规格条款均有对应的自动化测试验证。

## 2. 设计原则

1. **分层递进**: 按数据层 → 行为层 → 体验层顺序推进
2. **规格驱动**: 每个测试直接对应规格文档的验收条款
3. **黑盒验收**: 测试仅验证外部可观察行为，不依赖实现细节
4. **可维护性**: 遵循现有测试模式，复用测试基础设施

## 3. 分层测试设计

### 3.1 第一层：数据层测试（Rust 后端）

#### 3.1.1 按池筛选卡片查询

**规格来源**: card-note.md 5.2 - "系统 MUST 提供所属数据池筛选（含'全部池'语义）"

**测试文件**: `rust/tests/card_pool_filter_test.rs`

**测试场景**:
```rust
// Given 多个数据池，每个池有若干卡片
// When 按特定池 ID 查询卡片
// Then 结果仅包含该池的卡片（不含其他池卡片）
```

**验收标准**:
- [ ] `query_card_notes` API 支持可选的 `pool_id` 参数
- [ ] 未指定 `pool_id` 时返回所有池的卡片
- [ ] 指定 `pool_id` 时仅返回该池卡片
- [ ] 软删除卡片根据参数决定是否包含

#### 3.1.2 多成员协作一致性

**规格来源**: pool.md 6.2 - "任一成员对池内卡片笔记的增删改，系统 MUST 最终同步到其他成员可见状态"

**测试文件**: `rust/tests/pool_multi_member_sync_test.rs`

**测试场景**:
```rust
// Given 成员 A 和成员 B 在同一池
// When 成员 A 创建/修改/删除卡片
// Then 成员 B 最终可见与成员 A 一致的结果
```

**验收标准**:
- [ ] 成员 A 创建卡片后，成员 B 可查询到该卡片
- [ ] 成员 A 修改卡片后，成员 B 可见更新内容
- [ ] 成员 A 软删除卡片后，成员 B 默认列表不可见
- [ ] 同步延迟后状态最终一致（5秒内收敛）

**最终一致性定义**:
- 收敛超时阈值: 5秒
- 判定条件: 成员 B 查询结果与成员 A 本地状态一致
- 轮询间隔: 100ms，最多 50 次尝试

#### 3.1.3 并发幂等性

**规格来源**: pool.md 6.3 - "同一业务动作的重复提交 MUST 满足幂等预期"

**测试文件**: `rust/tests/pool_idempotency_test.rs`

**测试场景**:
```rust
// 场景 1: 重复加入申请
// Given 成员已提交加入申请
// When 重复提交相同加入申请
// Then 不会产生重复成员记录

// 场景 2: 重复审批
// Given 管理员已审批通过
// When 重复审批
// Then 不会产生副作用

// 场景 3: 并发修改冲突
// Given 成员 A 和成员 B 同时修改同一卡片
// When 两者提交修改
// Then 系统最终呈现一致结果（后提交者覆盖或合并成功）
```

**验收标准**:
- [ ] 重复 `join_by_code` 不产生重复成员
- [ ] 重复审批不产生状态错误
- [ ] 重复创建卡片不产生重复笔记
- [ ] 并发修改同一卡片最终呈现一致结果

### 3.2 第二层：行为层测试（Flutter 前端 + 集成）

#### 3.2.1 池筛选 UI 交互

**规格来源**: card-note.md 5.2 - 池筛选的 UI 交互验证

**测试文件**: `test/features/cards/cards_pool_filter_test.dart`

**测试场景**:
```dart
// Given 卡片页显示多个池的卡片
// When 用户选择特定池筛选器
// Then 列表仅显示该池卡片，UI 状态正确更新
```

**验收标准**:
- [ ] 筛选器组件可见且可交互
- [ ] 选择池后列表正确过滤
- [ ] "全部池"选项可重置筛选
- [ ] 筛选状态在页面切换后保持

### 3.3 第三层：体验层测试（质量属性）

#### 3.3.1 A11y 键盘导航

**规格来源**: ui-interaction.md 5.10.1 - "桌面端所有关键路径操作 MUST 可通过键盘完成"

**测试文件**: `test/features/accessibility/keyboard_navigation_test.dart`

**测试场景**:
```dart
// Given 桌面端卡片页
// When 仅使用键盘操作
// Then 可完成：导航、确认、取消、保存、返回
```

**验收标准**:
- [ ] Tab 键可遍历所有可交互元素
- [ ] Enter 键可触发默认动作
- [ ] Escape 键可取消/关闭弹窗
- [ ] 焦点顺序与视觉顺序一致
- [ ] 焦点可见性明确可辨

## 4. 依赖关系

```
第一层（数据层）
├── 3.1.1 按池筛选卡片查询
│   └── 依赖: query_card_notes API 扩展
├── 3.1.2 多成员协作一致性
│   └── 依赖: 3.1.1 完成
└── 3.1.3 并发幂等性
    └── 依赖: 3.1.2 完成

第二层（行为层）
└── 3.2.1 池筛选 UI 交互
    └── 依赖: 3.1.1 完成

第三层（体验层）
└── 3.3.1 A11y 键盘导航
    └── 依赖: 无
```

## 5. 实施顺序

1. **Phase 1**: 3.1.1 按池筛选卡片查询（基础数据能力）
2. **Phase 2**: 3.1.2 多成员协作一致性（核心协作能力）
3. **Phase 3**: 3.1.3 并发幂等性（稳定性保障）
4. **Phase 4**: 3.2.1 池筛选 UI 交互（前端集成）
5. **Phase 5**: 3.3.1 A11y 键盘导航（无障碍）

## 6. 验收标准汇总

### 6.1 最小验收集

- [ ] Given 笔记分布在多个池，When 按池筛选，Then 结果仅包含目标池笔记
- [ ] Given 池内成员修改笔记，When 同步收敛完成，Then 其他成员可见一致结果
- [ ] Given 重复提交加入申请，When 第二次提交，Then 不产生重复成员
- [ ] Given 桌面端关键页面，When 仅使用键盘，Then 可完成导航/确认/取消/保存

### 6.2 判定原则

1. 验收 MUST 以外部可观察行为为准
2. 验收依赖函数名、表结构、日志文本等实现细节为 FORBIDDEN

## 7. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 多成员测试需要模拟多个 endpoint | 中 | 使用测试替身模拟多成员场景 |
| API 扩展需要修改现有接口 | 中 | 使用可选参数保持向后兼容 |

## 8. 兼容与演进

1. 新增测试 MUST 遵循现有测试命名和结构约定
2. 测试失败时 MUST 提供明确的错误信息和恢复建议
3. 未来新增规格条款时，SHOULD 同步补充对应测试
