# 规格格式标准

## 新增需求

### 需求：主规格描述稳定行为

系统应在 `openspec/specs/` 下的规格中描述系统当前的稳定行为。
应避免在主规格中描述转换步骤或迁移历史。

#### 场景：编写主规格概述

- **操作**：创建或修订主规格
- **预期结果**：其概述应使用主动语态描述当前状态的行为
- **并且**：不应描述变更过程或迁移步骤

### 需求：主规格排除变更历史部分

系统不应在主规格中包含变更历史部分，如"核心变更"、"关键变更"或"行为变更"。

#### 场景：移除变更历史标记

- **操作**：主规格包含标记为"核心变更"、"关键变更"、"行为变更"或"转换"的标题或注释
- **预期结果**：这些部分应从主规格中移除
- **并且**：变更历史应记录在 `openspec/changes/<change-name>/specs/` 下的 delta spec 中

### 需求：主规格标题使用稳定命名

系统应将主规格标题命名为"<主题> 规格"。
不应在主规格标题中使用"转换"或"迁移"等措辞。

#### 场景：命名主规格

- **操作**：规格存储在 `openspec/specs/` 下
- **预期结果**：其标题应使用"规格"
- **并且**：不应包含"转换"或"迁移"

### 需求：Delta Spec 捕获变更叙述

系统应在 `openspec/changes/<change-name>/specs/` 下的 delta spec 中记录变更叙述和理由。

#### 场景：记录行为变更

- **操作**：变更引入新行为或移除旧行为
- **预期结果**：delta spec 应捕获变更叙述和理由
- **并且**：相应的主规格应更新为仅描述稳定的最终状态
